## 1. Node.js Fundamentals

### Q1: What is the Node.js event-driven architecture and how does the event loop work?

**A:** Node.js is built on an event-driven, single-threaded architecture. The event loop continuously checks for events and callbacks to execute:

- **Event Loop Phases:** timers → pending callbacks → idle/prepare → poll → check → close callbacks
- **Async I/O:** Uses libuv library for async I/O operations
- **Concurrency:** Single-threaded for JavaScript execution but uses worker threads for CPU-intensive operations

The event loop allows Node.js to handle thousands of concurrent connections efficiently without blocking operations.

### Q2: What is the difference between setImmediate() and setTimeout()?

**A:** Both are timer functions but execute at different phases of the event loop:

| Feature | setImmediate() | setTimeout() |
|---------|---|---|
| **Execution Phase** | Check phase (after I/O) | Timers phase |
| **Use Case** | After I/O operations | After specified delay |
| **Minimum Delay** | Next event loop iteration | ~4ms minimum |
| **Execution Order** | Executes first when called together | Executes after setImmediate |

```javascript
console.log('start');
setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
console.log('end');

// Output: start, end, immediate, timeout
```

### Q3: Explain Promises and async/await. What are their advantages?

**A:** 

**Promises:** Objects representing eventual completion of async operation with `.then()` and `.catch()` methods. Enable better control flow than callbacks.

```javascript
// Promise approach
fetchData()
  .then(data => processData(data))
  .then(result => saveResult(result))
  .catch(error => handleError(error));
```

**async/await:** Syntactic sugar over Promises. Makes async code look synchronous, easier to read and debug.

```javascript
// async/await approach
async function handleData() {
  try {
    const data = await fetchData();
    const result = await processData(data);
    await saveResult(result);
  } catch (error) {
    handleError(error);
  }
}
```

**Advantages:**
- Better error handling with try-catch
- Cleaner, more readable code
- Easier debugging (stack traces)
- Sequential async operations feel natural

### Q4: What is the purpose of middleware in Node.js?

**A:** Middleware is a function with access to request (req), response (res), and next function in the application request-response cycle.

**Common Uses:**
- Authentication and authorization checks
- Logging and monitoring requests
- Data validation and transformation
- Error handling
- CORS handling, compression, body parsing

```javascript
const authMiddleware = (req, res, next) => {
  if (req.headers.authorization) {
    req.user = verifyToken(req.headers.authorization);
    next();
  } else {
    res.status(401).send('Unauthorized');
  }
};
```

### Q5: What is callback hell and how do you avoid it?

**A:** Callback hell (pyramid of doom) occurs when callbacks are nested deeply, making code hard to read and maintain.

```javascript
// ❌ Callback Hell
getData((error, data) => {
  if (!error) {
    processData(data, (error, result) => {
      if (!error) {
        saveData(result, (error) => {
          if (!error) {
            console.log('Success');
          }
        });
      }
    });
  }
});

// ✅ Using Promises
getData()
  .then(data => processData(data))
  .then(result => saveData(result))
  .then(() => console.log('Success'))
  .catch(error => console.error(error));

// ✅ Using async/await
async function workflow() {
  try {
    const data = await getData();
    const result = await processData(data);
    await saveData(result);
    console.log('Success');
  } catch (error) {
    console.error(error);
  }
}
```

### Q6: What are streams in Node.js and their advantages?

**A:** Streams are objects that let you read data from source or write data to destination in chunks rather than loading everything into memory.

**Types:**
- **Readable:** Read data from source
- **Writable:** Write data to destination
- **Duplex:** Both readable and writable
- **Transform:** Modify data while reading/writing

**Advantages:**
- Memory efficient - process data in chunks
- Time efficient - start processing before all data received
- Perfect for large file handling, video/image processing

```javascript
// Reading large file efficiently with streams
fs.createReadStream('large-file.txt')
  .pipe(processStream)
  .pipe(fs.createWriteStream('output.txt'));
```

### Q9: Explain the Node.js event loop in detail with all phases

**A:** The event loop is the heart of Node.js. It has 6 phases executed in order:

```
┌───────────────────────────────┐
│           timers              │ setTimeout, setInterval
├───────────────────────────────┤
│     pending callbacks         │ Deferred I/O callbacks
├───────────────────────────────┤
│       idle, prepare           │ Internal use
├───────────────────────────────┤
│           poll                │ Retrieve new I/O events
├───────────────────────────────┤
│           check               │ setImmediate callbacks
├───────────────────────────────┤
│      close callbacks          │ Socket close handlers
└───────────────────────────────┘
```

**Phase Details:**

1. **Timers Phase:** Execute callbacks scheduled by setTimeout/setInterval
```javascript
setTimeout(() => console.log('timer'), 0);
```

2. **Pending Callbacks:** Execute deferred I/O callbacks
```javascript
fs.readFile('file.txt', (err, data) => {
  // Callback here
});
```

3. **Idle, Prepare:** Internal Node.js use only

4. **Poll Phase:** Wait for I/O events, longest phase
```javascript
// The event loop will wait here for new events
// Unless timers/setImmediate are waiting
```

5. **Check Phase:** Execute setImmediate callbacks
```javascript
setImmediate(() => console.log('immediate'));
```

6. **Close Callbacks:** Close event handlers
```javascript
socket.on('close', () => {
  // Executed here
});
```

**Execution Example:**
```javascript
console.log('start');

setTimeout(() => console.log('setTimeout'), 0);

Promise.resolve()
  .then(() => console.log('promise'));

setImmediate(() => console.log('setImmediate'));

console.log('end');

// Output:
// start
// end
// promise        (microtask queue - runs before next phase)
// setTimeout     (timers phase)
// setImmediate   (check phase)
```

**Microtask vs Macrotask:**
```javascript
// Microtasks (run before next macrotask phase)
// - Promises (.then, .catch)
// - process.nextTick()
// - MutationObserver
// - queueMicrotask()

// Macrotasks (one per phase)
// - setTimeout/setInterval
// - setImmediate
// - I/O operations
// - UI events

// process.nextTick runs before everything in current phase
process.nextTick(() => console.log('nextTick'));
Promise.resolve().then(() => console.log('promise'));
setTimeout(() => console.log('timer'), 0);

// Output: nextTick, promise, timer
```

### Q10: Explain Concurrency vs Parallelism in Node.js

**A:** These are often confused but are different concepts:

**Concurrency:**
- Multiple tasks executing in overlapping time periods
- Not necessarily at the same moment
- Single-threaded languages can be concurrent
- Node.js is concurrent through event loop

```javascript
// Concurrent execution (not parallel)
async function concurrentExample() {
  // Both start, but don't run simultaneously
  const [user, posts] = await Promise.all([
    fetchUser(),
    fetchPosts()
  ]);
  // Event loop switches between them
}

// Timeline:
// Start user fetch ─┐
//                  ├─ Event loop switches ─┐
// Start post fetch ┤                       ├─ Back to user
//                  └─ Event loop switches ─┤
//                                          └─ Both complete
```

**Parallelism:**
- Multiple tasks executing at the exact same moment
- Requires multiple cores/threads
- Node.js uses worker threads for parallelism

```javascript
// Parallel execution using Worker Threads
const { Worker } = require('worker_threads');

function parallelExample() {
  const worker = new Worker('./worker.js');
  
  // Main thread and worker thread run in parallel
  worker.on('message', (result) => {
    console.log('Result:', result);
  });
  
  worker.postMessage({ data: largeDataset });
}

// worker.js
const { parentPort } = require('worker_threads');

parentPort.on('message', (msg) => {
  const result = heavyComputation(msg.data);
  parentPort.postMessage(result);
});

// Timeline:
// Main Thread:    ├─ Task A ─┤ ├─ Task B ─┤ ├─ Task C ─┤
// Worker Thread:  ├─────── Heavy Compute ───────┤
// Both run simultaneously on different cores
```

**Concurrency vs Parallelism Comparison:**

| Aspect | Concurrency | Parallelism |
|--------|---|---|
| **Timing** | Overlapping, interleaved | Simultaneous execution |
| **Cores Needed** | Single core | Multiple cores |
| **Node.js Default** | Yes (event loop) | No (use Worker threads) |
| **Context Switching** | CPU switches | Native threading |
| **Example** | async/await | Worker threads |
| **For I/O** | Perfect | Overkill |
| **For CPU** | Bad | Good |

**When to Use:**

```javascript
// Use Concurrency (async/await) for I/O operations
async function getMultipleUsers() {
  // These run concurrently (not parallel)
  // But much faster than sequential
  return Promise.all([
    db.findUser(1),
    db.findUser(2),
    db.findUser(3)
  ]);
}

// Use Parallelism (Worker Threads) for CPU-intensive tasks
function processBigData() {
  const worker = new Worker('./processor.js');
  
  // Heavy computation runs in parallel
  worker.postMessage(largeDataset);
  
  worker.on('message', result => {
    // Main thread still responsive
  });
}
```

## 2. Express.js Framework

### Q1: What is Express.js and what problems does it solve?

**A:** Express.js is a lightweight web application framework for Node.js that simplifies building web servers and APIs.

**Problems Solved:**
- Simplifies HTTP request/response handling
- Provides routing mechanism for different URLs
- Middleware support for code reusability
- Template engine integration for rendering views
- Built-in utilities for common tasks (parsing, compression, etc.)

### Q2: Explain routing in Express. What are route parameters, query parameters, and body parameters?

**A:**

```javascript
// Route Parameters: Part of URL path
app.get('/users/:id', (req, res) => {
  console.log(req.params.id); // Access route parameter
});

// Query Parameters: After ? in URL
// GET /users?page=1&limit=10
app.get('/users', (req, res) => {
  console.log(req.query.page, req.query.limit);
});

// Body Parameters: In request body
app.post('/users', (req, res) => {
  console.log(req.body); // Requires body-parser middleware
});
```

### Q3: What is the difference between app.use() and route handlers like app.get()?

**A:**

| Feature | app.use() | app.get() |
|---------|-----------|-----------|
| **Type** | Middleware | Route handler |
| **Methods** | All HTTP methods | Specific (GET) |
| **Routing** | Matches path prefix | Exact path match |
| **Use Case** | Logging, auth, parsing | Handling specific endpoint |

```javascript
// Runs for all routes and methods
app.use(authMiddleware);

// Runs only for GET /api/users
app.get('/api/users', (req, res) => {
  res.json(users);
});
```

### Q4: How do you handle errors in Express?

**A:** Error handling approaches:

```javascript
// Try-catch in async route handlers
app.get('/users/:id', async (req, res, next) => {
  try {
    const user = await User.findById(req.params.id);
    res.json(user);
  } catch (error) {
    next(error); // Pass to error handler
  }

// Error handling middleware (4 parameters)

  res.status(err.status || 500).json({

// Custom error class
class ValidationError extends Error {
  constructor(message) {

    this.status = 400;
  }
}
```

## 3. SQL & Relational Databases

### Q1: What is SQL and what are the main SQL statements?

**A:** SQL (Structured Query Language) is used to manage relational databases.

**Main Categories:**

**DDL (Data Definition Language):**
```sql
-- CREATE: Define new database objects
CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(100) UNIQUE,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- ALTER: Modify existing objects
ALTER TABLE users ADD COLUMN phone VARCHAR(20);

-- DROP: Delete objects
DROP TABLE users;
```

**DML (Data Manipulation Language):**
```sql
-- INSERT: Add data
INSERT INTO users (name, email) VALUES ('John', 'john@example.com');

-- SELECT: Retrieve data
SELECT * FROM users WHERE age > 18;

-- UPDATE: Modify data
UPDATE users SET age = 25 WHERE id = 1;

-- DELETE: Remove data
DELETE FROM users WHERE id = 1;
```

**DCL (Data Control Language):**
```sql
-- GRANT: Give permissions
GRANT SELECT, INSERT ON users TO user@localhost;

-- REVOKE: Remove permissions
REVOKE DELETE ON users FROM user@localhost;
```

**TCL (Transaction Control Language):**
```sql
-- COMMIT: Save changes
COMMIT;

-- ROLLBACK: Undo changes
ROLLBACK;

-- SAVEPOINT: Create checkpoint
SAVEPOINT sp1;
ROLLBACK TO sp1;
```

### Q2: What are JOINs and their types?

**A:** JOINs combine rows from multiple tables based on related columns.

```sql
-- Sample Data:
-- users: [1->John, 2->Jane, 3->Bob]
-- orders: [1->user1, 2->user1, 3->user2]

-- INNER JOIN: Only matching rows from both tables
SELECT u.name, o.id
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
-- Result: John (2 orders), Jane (1 order)

-- LEFT JOIN: All from left table + matching from right
SELECT u.name, COUNT(o.id) as orders
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id;
-- Result: John (2), Jane (1), Bob (0)

-- RIGHT JOIN: All from right table + matching from left
SELECT u.name, o.id
FROM users u
RIGHT JOIN orders o ON u.id = o.user_id;
-- Result: Orders that exist (may include NULL users)

-- FULL OUTER JOIN: All rows from both tables
SELECT u.name, o.id
FROM users u
FULL OUTER JOIN orders o ON u.id = o.user_id;
-- Result: All users and all orders

-- CROSS JOIN: Cartesian product (all combinations)
SELECT u.name, p.name
FROM users u
CROSS JOIN products p;
-- Result: Every user paired with every product
```

**Visual Representation:**
```
Table A    INNER JOIN    Table B
  ╔═╗                      ╔═╗
  ║ ║ A∩B only            ║ ║
  ╚═╝                      ╚═╝

Table A    LEFT JOIN     Table B
  ╔═╗                      ╔═╗
  ║ ║ All A + matched B   ║ ║
  ╚═╝                      ╚═╝

Table A    FULL JOIN     Table B
  ╔═╗                      ╔═╗
  ║ ║ All A + All B       ║ ║
  ╚═╝                      ╚═╝
```

### Q3: What are indexes and how do they improve performance?

**A:** Indexes create sorted data structures for faster lookups.

```sql
-- Create index on single column
CREATE INDEX idx_email ON users(email);

-- Create composite index (multiple columns)
CREATE INDEX idx_user_date ON orders(user_id, created_at);

-- Create unique index
CREATE UNIQUE INDEX idx_unique_email ON users(email);

-- Query execution with index:
-- Without index: Full table scan O(n)
-- With index: B-tree search O(log n)

SELECT * FROM users WHERE email = 'john@example.com';
-- Without index: Scans all 1,000,000 rows
-- With index: Scans ~20 rows (log2(1,000,000) ≈ 20)

-- Drop index
DROP INDEX idx_email ON users;
```

**Index Trade-offs:**

| Benefit | Cost |
|---------|------|
| Faster SELECT | Slower INSERT/UPDATE/DELETE |
| Faster sorting | Extra disk space |
| Faster filtering | Memory overhead |

```sql
-- Check index usage
EXPLAIN SELECT * FROM users WHERE email = 'john@example.com';
-- Shows if index is used in query plan
```

### Q4: What are aggregate functions and GROUP BY?

**A:** Aggregate functions compute values across rows.

```sql
-- Basic aggregate functions
SELECT 
  COUNT(*) as total_users,           -- Count rows
  COUNT(DISTINCT status) as statuses, -- Count unique values
  SUM(amount) as total_amount,       -- Sum values
  AVG(age) as average_age,           -- Average
  MIN(created_at) as oldest,         -- Minimum
  MAX(created_at) as newest          -- Maximum
FROM users;

-- GROUP BY: Group rows by column values
SELECT 
  status,
  COUNT(*) as user_count,
  AVG(age) as avg_age
FROM users
GROUP BY status;
-- Result:
-- active    | 5000 | 28.5
-- inactive  | 2000 | 32.1

-- HAVING: Filter grouped results
SELECT 
  department,
  COUNT(*) as emp_count,
  AVG(salary) as avg_salary
FROM employees
GROUP BY department
HAVING AVG(salary) > 50000;  -- Only departments with avg salary > 50k
```

### Q5: What is normalization and why is it important?

**A:** Normalization organizes data to reduce redundancy and dependency.

**Normal Forms:**

**1NF (First Normal Form):**
- Each cell contains single value (no arrays in database)
- No duplicate columns

```sql
-- ❌ Not 1NF (skills array)
CREATE TABLE users (
  id INT,
  name VARCHAR(100),
  skills VARCHAR(100)  -- Contains: "JavaScript, Python, Java"
);

-- ✅ 1NF (separate table for skills)
CREATE TABLE user_skills (
  user_id INT,
  skill VARCHAR(100)
);
```

**2NF (Second Normal Form):**
- Must be 1NF
- All non-key columns depend on entire primary key

```sql
-- ❌ Not 2NF (professor name depends only on prof_id, not student_id)
CREATE TABLE enrollments (
  student_id INT,
  course_id INT,
  professor_id INT,
  professor_name VARCHAR(100),  -- Shouldn't be here
  PRIMARY KEY (student_id, course_id)
);

-- ✅ 2NF
CREATE TABLE courses (
  course_id INT PRIMARY KEY,
  professor_id INT,
  professor_name VARCHAR(100)
);

CREATE TABLE enrollments (
  student_id INT,
  course_id INT,
  PRIMARY KEY (student_id, course_id)
);
```

**3NF (Third Normal Form):**
- Must be 2NF
- Non-key columns depend only on primary key, not on each other

```sql
-- ❌ Not 3NF (city depends on country, not directly on user_id)
CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  country_id INT,
  country_name VARCHAR(100),  -- Transitive dependency
  city_name VARCHAR(100)
);

-- ✅ 3NF
CREATE TABLE countries (
  id INT PRIMARY KEY,
  name VARCHAR(100)
);

CREATE TABLE users (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  country_id INT FOREIGN KEY
);
```

### Q6: What are UNION, INTERSECT, and EXCEPT?

**A:** Set operations combine results from multiple queries.

```sql
-- UNION: Combine results, remove duplicates
SELECT email FROM users WHERE status = 'active'
UNION
SELECT email FROM newsletter_subscribers;
-- Returns unique emails from both queries

-- UNION ALL: Combine results, keep duplicates
SELECT email FROM users
UNION ALL
SELECT email FROM newsletter_subscribers;
-- May include same email twice

-- INTERSECT: Return only common rows
SELECT email FROM users
INTERSECT
SELECT email FROM newsletter_subscribers;
-- Returns emails that are in BOTH tables

-- EXCEPT: Return rows in first query but not in second
SELECT email FROM users
EXCEPT
SELECT email FROM deleted_accounts;
-- Returns emails not in deleted_accounts
```

### Q7: What are subqueries and correlated subqueries?

**A:** Subqueries are queries nested inside other queries.

```sql
-- Simple subquery
SELECT name, salary FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);
-- Find employees earning above average

-- Subquery in WHERE clause
SELECT * FROM users
WHERE country_id IN (
  SELECT id FROM countries WHERE continent = 'Asia'
);

-- Subquery in FROM clause (derived table)
SELECT country, avg_salary FROM (
  SELECT country_id, AVG(salary) as avg_salary
  FROM employees
  GROUP BY country_id
) as country_salaries;

-- Correlated subquery (references outer query)
SELECT e.name, e.salary
FROM employees e
WHERE e.salary > (
  SELECT AVG(salary) 
  FROM employees 
  WHERE department_id = e.department_id
);
-- Find employees earning above their department average
-- Subquery runs for EACH row in outer query (slower)

-- Performance: Use JOIN instead of correlated subquery
SELECT e.name, e.salary
FROM employees e
JOIN (
  SELECT department_id, AVG(salary) as avg_sal
  FROM employees
  GROUP BY department_id
) as dept_avg ON e.department_id = dept_avg.department_id
WHERE e.salary > dept_avg.avg_sal;
```

## 4. MongoDB & Databases

### Q1: What is MongoDB and how is it different from SQL databases?

**A:**

| Feature | MongoDB (NoSQL) | SQL Databases |
|---------|---|---|
| **Schema** | Flexible - different structures | Fixed schema |
| **Data Type** | JSON-like documents | Tables with columns |
| **Scalability** | Horizontal (sharding) | Vertical |
| **Relationships** | Embedded documents | Foreign keys & JOINs |
| **ACID** | Multi-doc transactions (v4.0+) | Full ACID compliance |

**When to use MongoDB:**
- Unstructured/semi-structured data
- Rapid development with changing schema
- Need for horizontal scaling
- Document-oriented workflows

### Q2: Explain the CRUD operations in MongoDB

**A:**

```javascript
// Create
await User.insertOne({ name: 'John', age: 25 });
await User.insertMany([{ name: 'Jane' }, { name: 'Bob' }]);

// Read
await User.find({ age: { $gt: 20 } });
await User.findOne({ email: 'john@example.com' });

// Update
await User.updateOne({ _id: 123 }, { $set: { age: 26 } });
await User.updateMany({ status: 'inactive' }, { $set: { status: 'active' } });
await User.replaceOne({ _id: 123 }, { name: 'Jane', age: 30 });

// Delete
await User.deleteOne({ _id: 123 });
await User.deleteMany({ status: 'deleted' });
```

### Q6: What are transactions in MongoDB and when would you use them?

### Q8: Explain ACID Properties in Databases

**A:** ACID ensures reliable database transactions:

#### **A - Atomicity**
All or nothing - transaction either fully completes or fully fails.

```javascript
// Example: Money transfer between accounts
// EITHER both succeed OR both fail, never partial

const session = await db.startSession();
session.startTransaction();

try {
  // Debit account A
  await Account.updateOne(
    { _id: accountA },
    { $inc: { balance: -1000 } },
    { session }
  );
  
  // Credit account B
  await Account.updateOne(
    { _id: accountB },
    { $inc: { balance: 1000 } },
    { session }
  );
  
  await session.commitTransaction();
} catch (error) {
  // If any operation fails, BOTH are rolled back
  await session.abortTransaction();
  throw error;
}

// Before: A=$1000, B=$2000
// Success: A=$0, B=$3000
// Failure: A=$1000, B=$2000 (both unchanged)
```

#### **C - Consistency**
Database goes from one valid state to another valid state.

```javascript
// Example: Inventory management
// Total inventory must always equal sum of all warehouse quantities

// ❌ Inconsistent state (violates business rules)
// Total: 100, Warehouse A: 30, Warehouse B: 50, Warehouse C: 10
// Sum: 30 + 50 + 10 = 90 ≠ 100

// ✅ Consistent state
// Total: 100, Warehouse A: 30, Warehouse B: 50, Warehouse C: 20
// Sum: 30 + 50 + 20 = 100

// Database constraints ensure consistency:
userSchema.pre('save', async function() {
  const total = this.experience + this.projects;
  if (total > this.maxAllowed) {
    throw new Error('Consistency violation');
  }
});
```

#### **I - Isolation**
Concurrent transactions don't interfere with each other.

```javascript
// Example: Two transactions at same time

// Transaction 1: Read user's balance
const balance1 = await Account.findById(userId);
console.log('Balance:', balance1);

// Transaction 2 (simultaneous): Withdraw money
await Account.updateOne(
  { _id: userId },
  { $inc: { balance: -500 } }
);

// With proper isolation, Transaction 1 either:
// - Sees balance BEFORE withdraw (Read Uncommitted)
// - Sees balance AFTER withdraw (Read Committed - default)
// - Never sees intermediate state (Repeatable Read)
// - Completely isolated (Serializable)

// Isolation Levels:
// Read Uncommitted: Dirty reads possible
// Read Committed: No dirty reads
// Repeatable Read: Phantom reads possible
// Serializable: Complete isolation (slowest)
```

#### **D - Durability**
Once committed, data survives any failure.

```javascript
// Example: Disk failure shouldn't lose committed data

await db.transaction(async (tx) => {
  await tx.users.create({ name: 'John' });
  // Data is logged to disk
  
  // Even if server crashes now:
  // - Data is recovered from logs on restart
  // - Transaction is either fully replayed or fully rolled back
});

// Durability mechanisms:
// - Write-ahead logging (WAL)
// - Disk sync after commit
// - Replication to other servers
```

**ACID in Different Databases:**

| Property | SQL | MongoDB | NoSQL |
|----------|-----|---------|-------|
| **Atomicity** | Single row | Multi-doc (4.0+) | Limited |
| **Consistency** | Enforced | Schema validation | Application level |
| **Isolation** | Configurable levels | Snapshot isolation | Limited |
| **Durability** | Guaranteed | Configurable | Varies |

**Trade-offs:**

```javascript
// ACID is slower but safer
// - Write-ahead logging overhead
// - Lock contention
// - Coordination overhead

// For high-speed, distributed systems:
// - Eventual consistency (NoSQL)
// - Per-document atomicity
// - Acceptance of temporary inconsistency

// Example: Social media likes
// 100 million likes/second is impossible with full ACID
// Solution: Eventual consistency
// - Count likes distributed globally
// - Sync every few seconds
// - Users see slightly stale counts
```

### Q2: Explain JWT (JSON Web Tokens) and how it works

**Multipart Upload:**
```javascript
// Upload large file in chunks
const fs = require('fs');

async function multipartUpload() {
  const fileSize = fs.statSync('large-file.zip').size;
  const partSize = 5 * 1024 * 1024; // 5MB per part
  
  // Initiate multipart upload
  const multipart = await s3.createMultipartUpload({
    Bucket: 'my-bucket',
    Key: 'large-file.zip'
  }).promise();
  
  const uploadId = multipart.UploadId;
  const parts = [];
  
  // Upload each part
  for (let i = 1; i <= Math.ceil(fileSize / partSize); i++) {
    const start = (i - 1) * partSize;
    const end = Math.min(start + partSize, fileSize);
    
    const chunk = fs.readFileSync('large-file.zip', {
      start,
      end: end - 1
    });
    
    const part = await s3.uploadPart({
      Bucket: 'my-bucket',
      Key: 'large-file.zip',
      PartNumber: i,
      UploadId: uploadId,
      Body: chunk
    }).promise();
    
    parts.push({
      PartNumber: i,
      ETag: part.ETag
    });
  }
  
  // Complete multipart upload
  const result = await s3.completeMultipartUpload({
    Bucket: 'my-bucket',
    Key: 'large-file.zip',
    UploadId: uploadId,
    MultipartUpload: { Parts: parts }
  }).promise();
  
  console.log('Upload complete:', result.Location);
}

// Benefits:
// - Resume failed uploads (retry individual parts)
// - Upload in parallel (faster)
// - Reduce timeout risk
// - More reliable for large files
```

### Q4: What is caching and caching strategies?

**A:**

```javascript
// Cache-Aside (Lazy Loading)

  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);
  
  const user = await db.users.findById(id);

// Write-Through

  const user = await db.users.create(userData);

};
```

```javascript

const posts = await Post.find();
for (let post of posts) {
  post.author = await User.findById(post.authorId); // N+1 queries
}

// ✅ Solution 1: JOIN query
const posts = await Post.find().populate('author');

// ✅ Solution 2: Batch load
const posts = await Post.find();
const authorIds = posts.map(p => p.authorId);
const authors = await User.find({ _id: { $in: authorIds } });
const authorMap = new Map(authors.map(a => [a._id, a]));
posts.forEach(p => p.author = authorMap.get(p.authorId));
```
