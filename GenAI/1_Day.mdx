## Topic 1: What is Generative AI?

### Understanding the Fundamental Difference

#### Traditional Software (Deterministic)

```
Input → Logic/Rules → Output (always the same)
```

**Example:**

```javascript
function add(a, b) {
  return a + b;
}
add(5, 3) // Always returns 8
```

Every time you input 5 and 3, you get 8. There's logic, rules, and determinism.

#### Generative AI (Non-deterministic)

```
Input (Prompt) → Neural Network → Generated Output (can vary)
```

**Example:**

```
Prompt: "Write a poem about winter"

Output (Run 1):
"Snow falls gently, blanketing the earth..."

Output (Run 2):
"Winter's frost paints the world white..."

Output (Run 3):
"Ice crystals dance in December's breeze..."
```

Same input, different outputs. The model **generates** new content.

### What Can Generative AI Create?

- **Text**: Articles, emails, summaries, code
- **Code**: Functions, entire applications
- **Images**: From text descriptions (DALL-E, Midjourney)
- **Audio**: Voice synthesis, music
- **Video**: Animations, deepfakes

### Real Companies Using This

| Company | Products |
|---------|----------|
| OpenAI | GPT-4, GPT-3.5, ChatGPT |
| Google | Gemini, Bard, PaLM |
| Anthropic | Claude |
| Meta | Llama |
| Stability AI | Stable Diffusion |

### Key Insight for Interviews

> **Generative AI doesn't have a "database" of answers. It doesn't retrieve facts. It **predicts** what the next word/token should be based on patterns learned during training.**

## Topic 2: What is an LLM? (Deep Dive)

### LLM = Large Language Model

**Breaking it down:**

- **Large**: Billions of parameters (think: neurons in a neural network). GPT-3 has 175 billion parameters.
- **Language**: Trained on text data (books, websites, code, etc.)
- **Model**: A mathematical function that takes input and produces output

### How Does an LLM Actually Work?

**Simplified Mental Model:**

Imagine you're playing a word prediction game:

```
You read: "The sky is..."
You guess: "blue"

But you've read 10 billion examples of sentences.
You've seen patterns like:
- "The sky is blue" (50 million times)
- "The sky is clear" (30 million times)
- "The sky is cloudy" (20 million times)

Based on frequency and context, you predict the next word.
```

An LLM does exactly this, but with **mathematical precision** using neural networks.

### The LLM is NOT

<details>
<summary>What LLMs Are NOT</summary>

❌ A database that looks up answers  
❌ Conscious or intelligent  
❌ Certain about anything  
❌ Connected to the internet (unless specifically built to be)  
❌ Always factually correct  

</details>

### The LLM IS

<details>
<summary>What LLMs ARE</summary>

✅ A statistical model trained on patterns  
✅ Very good at predicting plausible text  
✅ Probabilistic (outputs probabilities for each possible next token)  
✅ Constrained by what it was trained on  
✅ Fast (can generate tokens in milliseconds)  

</details>

### Real Example: How ChatGPT Generates an Answer

**Your Prompt:**
```
What is React?
```

**What Actually Happens Inside:**

1. Your prompt is converted to tokens
2. The model processes each token through layers of neural networks
3. At the end, it calculates probabilities for the next token:
   ```
   React -> 45% probability
   is -> 30% probability
   ...a -> 15% probability
   ```
4. It picks "React" (highest probability) or randomly based on temperature
5. Now it has: "What is React"
6. It repeats this process for the next token
7. It continues until it hits a stop condition (end token, max length, etc.)

**Output:**
```
React is a JavaScript library developed by Meta for building user interfaces...
```

### Interview Answer

**Q: How does ChatGPT know my question?**

**A:** It doesn't "know" it in a database sense. It reads your prompt as tokens, processes them through a neural network trained on billions of examples, and predicts the most statistically likely next tokens to generate a coherent response.

## Topic 3: Tokens (Critical Concept)

### What is a Token?

A token is the smallest unit of text that an LLM understands.

**NOT a word. NOT a character.**

A token is typically:
- A word
- A punctuation mark
- Part of a word
- A whitespace

### Real Examples

```
"Hello" → 1 token ([15496])
"Hello World" → 2 tokens ([15496, 2159])
"Hello, World!" → 4 tokens ([15496, 11, 2159, 0])
"it's" → 2 tokens ([270, 311]) # "it" + "'s"
```

**This is important**: "it's" is 2 tokens, not 1 word.

### Why Are Tokens Crucial?

#### 1. Cost Calculation

```
GPT-4 Pricing (as of 2024):
- Input: $0.03 per 1,000 tokens
- Output: $0.06 per 1,000 tokens

Your prompt: "Explain React" = ~2 tokens
Model response: "React is..." (500 tokens)

Cost = (2 * 0.03 + 500 * 0.06) / 1000 = $0.00303
```

#### 2. Context Window Limits

```
Claude 3.5 Sonnet: 200,000 token context window

If you upload a document:
- 50,000 tokens (document)
- 100 tokens (your question)
- You have 149,900 tokens left for the response

Exceed this, and the model can't process it.
```

#### 3. Latency

More tokens = slower response (roughly linear)

### How to Estimate Tokens?

**Rule of thumb:**

```
1 English word ≈ 1.3 tokens
1 line of code ≈ 2-4 tokens

100 words ≈ 130 tokens
1 page of text ≈ 500 tokens
```

### For Developers

You can check token count programmatically:

```javascript
const { encoding_for_model } = require("js-tiktoken");

const enc = encoding_for_model("gpt-4");
const tokens = enc.encode("Hello World");
console.log(tokens.length); // 2
```

### Interview Answer

**Q: Why do token limits matter?**

**A:** Because context window (model memory) is measured in tokens. If your prompt + document + conversation history exceeds the context window, the model can't process it. Also, pricing is based on tokens, so understanding token count is essential for cost optimization.

## Topic 4: Context Window (Your Token Budget)

### What is a Context Window?

The **maximum number of tokens** a model can process in a single API call.

Think of it as your "working memory" for the model.

### Context Window Examples (2024)

```
GPT-3.5:  4,096 tokens
GPT-4:    8,192 tokens (standard)
GPT-4:    128,000 tokens (extended)
Claude 3.5: 200,000 tokens
Gemini:    32,000 tokens (varies by model)
```

### What Fits Inside Context?

```
[Input Tokens] = [System Prompt] + [Previous Conversation] + [Documents] + [Your Question]
```

**Example Scenario:**

```
System Prompt:     500 tokens
("You are an expert Python developer...")

Previous Conversation:
- User: "How do I use decorators?"  (50 tokens)
- Assistant: "Decorators are..." (300 tokens)
- User: "Give me an example" (20 tokens)
- Assistant: "Here's a decorator example..." (250 tokens)
= 620 tokens

Your Document (PDF):
"Our company API documentation..." (15,000 tokens)

Your New Question:
"How do I authenticate?" (15 tokens)

Total: 500 + 620 + 15,000 + 15 = 16,135 tokens
```

**If context window is 8,192 tokens:**
❌ This won't fit! You'd get an error.

### The Problem This Creates

**Q: Can I upload my entire company database to ChatGPT?**

**A: No, because:**

1. Databases are millions of tokens
2. Context window is fixed (e.g., 128k tokens)
3. You'd have to split it into multiple requests
4. This is inefficient and expensive

**This is why RAG was invented.**

### Practical Implications

**For Developers:**

```javascript
// When making API calls, track tokens:
const { encoding_for_model } = require("js-tiktoken");
const enc = encoding_for_model("gpt-4");

let totalTokens = 0;

// Count system prompt
totalTokens += enc.encode(systemPrompt).length;

// Count conversation history
conversation.forEach(msg => {
  totalTokens += enc.encode(msg.content).length;
});

// Check if you have room for response
const maxContextWindow = 8192;
const tokensForResponse = maxContextWindow - totalTokens;

if (tokensForResponse < 500) {
  console.warn("Warning: Limited space for response");
}
```

### Interview Answer

**Q: Why is context window important?**

**A:** Context window is your model's "working memory." It limits how much information you can give the model in one request. If you exceed it, the request fails. For applications handling large documents, you need strategies like RAG (Retrieval-Augmented Generation) to work around context limits.

## Topic 5: Temperature (Controlling Randomness)

### What is Temperature?

A parameter that controls **how random/creative** the model's responses are.

```
Range: 0.0 to 2.0 (varies by model)
```

### Temperature = 0 (Deterministic)

```
Setting: temperature = 0

Behavior:
- Always picks the most likely token
- Responses are consistent/reproducible
- No randomness
```

**Example:**

```
Prompt: "2 + 2 = ?"

Response (Run 1): "4"
Response (Run 2): "4"
Response (Run 3): "4"
```

**When to use:**
- Data extraction
- Coding problems
- Factual questions
- Customer service (consistency matters)

### Temperature = 1 (Balanced)

```
Setting: temperature = 1

Behavior:
- Balanced between likelihood and creativity
- Responses vary but stay coherent
- Default for most applications
```

**Example:**

```
Prompt: "Write a haiku about rain"

Response (Run 1): "Raindrops fall gently, soaking the thirsty green earth..."
Response (Run 2): "Gray clouds open up, drenching the world below..."
Response (Run 3): "Water falls from clouds, nourishing all life on earth..."
```

**When to use:**
- General conversation
- Content creation
- Creative writing
- Most production applications

### Temperature = 1.5+ (Very Creative)

```
Setting: temperature = 1.8

Behavior:
- Very random and unpredictable
- Can be nonsensical
- Highly varied responses
```

**Example:**

```
Prompt: "Tell me about a magical forest"

Response (Run 1): "The forest whispers secrets of ancient times..."
Response (Run 2): "Kaleidoscopic trees pulse with forgotten melodies..."
Response (Run 3): "Crystalline shadows dance where moonbeams sing..."
```

**When to use:**
- Brainstorming
- Creative fiction
- Ideation sessions
- Art generation

### How Temperature Works Mathematically

```
Model generates probabilities for next token:

Token     Probability
"the"     0.5
"a"       0.3
"this"    0.15
"one"     0.05

With temperature=0:
→ Always pick "the" (highest)

With temperature=1:
→ Sample according to probabilities
→ "the" chosen 50% of the time

With temperature=2:
→ Flatten the distribution
→ All tokens more equally likely
→ "one" might be chosen more often
```

### Visual Representation

```
Temperature = 0 (Sharp)
████████████░░░░░░░░░░░░░░
"the" heavily favored

Temperature = 1 (Normal)
███████░░░░░░░░░░░░░░░░░░░░
Balanced distribution

Temperature = 1.8 (Flat)
████░░░░░░░░░░░░░░░░░░░░░░░
All tokens more equal
```

### Practical Code Example

```javascript
const response = await openai.chat.completions.create({
  model: "gpt-4",
  messages: [{ role: "user", content: "Write a story" }],
  temperature: 0.7, // Slightly creative
});

// For coding: temperature = 0
// For creative writing: temperature = 1.2
// For classification: temperature = 0
```

### Interview Answer

**Q: How does temperature affect LLM outputs?**

**A:** Temperature controls randomness. At 0, the model always picks the most likely token (deterministic, good for coding). At 1+, it becomes more creative and unpredictable. Higher temperatures introduce more variation but risk incoherence. The choice depends on your use case—factual tasks use low temperature, creative tasks use high.

## Topic 6: Prompt Engineering (Deep Dive)

### What is Prompt Engineering?

The art and science of **writing prompts that get better responses** from LLMs.

### Poor Prompt vs Better Prompt

#### ❌ Poor Prompt:

```
Explain React
```

**Problems:**
- Vague and generic
- No context about the audience
- No indication of desired depth
- Model doesn't know your expertise level

#### ✅ Better Prompt:

```
Act as a Senior Frontend Engineer with 10 years of experience.

Explain React Fiber (React's reconciliation engine) to a developer 
who has 3 years of experience building React applications.

Include:
1. What problem it solves
2. How it differs from the old reconciliation algorithm
3. A real-world code example
4. When/why a developer should care

Keep it practical, not academic.
```

**Why it's better:**
- Clear role (Senior Frontend Engineer)
- Specific audience (3 years experience)
- Context window (React Fiber, not React basics)
- Structured output (numbered points)
- Tone guidance (practical, not academic)

### Prompt Engineering Techniques

#### 1. Zero-Shot Prompting

No examples. Just ask directly.

```
Classify this text as positive or negative:
"I love this product! Best purchase ever!"
```

**Result:** "Positive"

**When to use:**
- Simple, straightforward tasks
- Tasks the model was directly trained on

#### 2. Few-Shot Prompting

Provide examples before asking.

```
Classify sentiment:

Text: "This movie was amazing!"
Sentiment: Positive

Text: "Terrible experience, never again."
Sentiment: Negative

Text: "It was okay, nothing special."
Sentiment: Neutral

Now classify:
Text: "I'm so happy with my purchase!"
Sentiment: ?
```

**Result:** "Positive"

**Why it works:**
- Model learns from examples
- Establishes the pattern you want
- Better accuracy on complex tasks

**When to use:**
- Complex classification
- Specific formatting requirements
- Tasks outside training data

#### 3. Chain-of-Thought Prompting

Ask the model to explain its reasoning step-by-step.

```
❌ Weak:
Question: If there are 3 apples and you remove 2, how many are left?
Answer: ?

✅ Strong:
Question: If there are 3 apples and you remove 2, how many are left?
Think step by step.

Answer:
- Start with 3 apples
- Remove 2
- Calculate: 3 - 2 = 1
- Final answer: 1 apple remains
```

**When to use:**
- Math problems
- Logic puzzles
- Complex reasoning tasks

**Result:** Significantly higher accuracy.

#### 4. Role-Based Prompting

Give the model a role/persona.

```
❌ Weak:
What should I eat for lunch?

✅ Strong:
You are a professional nutritionist with 20 years of experience.
A client just asked: "What should I eat for lunch?"

Provide:
1. A balanced meal recommendation
2. Nutritional breakdown
3. Why this choice is healthy
4. Alternatives for dietary restrictions
```

**Result:** Much more detailed, professional response.

#### 5. Structured Output

Ask for specific format.

```
❌ Weak:
Tell me about JavaScript async/await.

✅ Strong:
Explain JavaScript async/await. Format your response as:

{
  "definition": "...",
  "use_cases": ["...", "..."],
  "code_example": "...",
  "common_mistakes": ["...", "..."]
}
```

**Result:** Structured JSON you can parse programmatically.

#### 6. Negative Prompting

Tell the model what NOT to do.

```
❌ Weak:
Write a product description for a laptop.

✅ Strong:
Write a product description for a laptop.

DO NOT:
- Use marketing jargon like "revolutionary"
- Make false claims about performance
- Be longer than 3 sentences
- Mention price

DO:
- Be honest about specs
- Use simple language
- Highlight real benefits
```

**Result:** More controlled, focused output.

### Real-World Prompt Example

```
You are an experienced code reviewer.

A junior developer submitted this code:

```python
def fibonacci(n):
    if n <= 1:
        return n
    else:
        return fibonacci(n-1) + fibonacci(n-2)
```

Provide a code review that includes:
1. What the code does correctly
2. Performance issues (with explanation)
3. An optimized solution
4. Why the optimization works
5. What the junior developer should learn from this

Be constructive and educational, not condescending.
```

### Interview Answer

**Q: What is prompt engineering?**

**A:** Prompt engineering is the practice of writing effective prompts to get better outputs from LLMs. Key techniques include:
- **Zero-shot**: Ask directly without examples
- **Few-shot**: Provide examples before asking
- **Chain-of-thought**: Ask for step-by-step reasoning
- **Role-based**: Give the model a persona
- **Structured output**: Request specific formats

The better your prompt, the better your result.

## Topic 7: System Prompt vs User Prompt

### Architecture

```
┌─────────────────────────────┐
│    SYSTEM PROMPT            │  (Sets behavior/rules)
├─────────────────────────────┤
│    USER PROMPT              │  (Actual question/task)
├─────────────────────────────┤
│         LLM                 │
├─────────────────────────────┤
│    GENERATED RESPONSE       │
└─────────────────────────────┘
```

### System Prompt

**Definition:** A prompt that sets the behavior, rules, and personality of the LLM.

**Important:** The user usually doesn't see or control this.

### User Prompt

**Definition:** The actual question or request from the user.

**Important:** This is what the user inputs.

### Real Example

**System Prompt:**
```
You are Claude, an AI assistant made by Anthropic.
You are helpful, harmless, and honest.
You should:
- Provide accurate information
- Decline harmful requests politely
- Be concise unless asked otherwise
- Acknowledge uncertainty when appropriate
```

**User Prompt:**
```
How do I learn React?
```

**Combined Effect:**
The model now:
- Responds as "Claude"
- Follows Anthropic's values
- Answers the user's question helpfully

### Another Example: ChatBot

**System Prompt:**
```
You are a customer service representative for TechStore.
You have access to these tools:
- Check order status
- Process returns
- Answer FAQ

Be friendly, professional, and solve problems quickly.
If asked about something outside your scope, 
politely escalate to a human agent.
```

**User Prompt:**
```
My order hasn't arrived in 2 weeks. Order #12345.
```

**How It Works:**
1. System prompt tells it to be helpful and professional
2. System prompt gives it available tools
3. User asks about an order
4. Model checks order status tool
5. Model provides friendly, professional response

### System Prompt vs User Prompt Comparison

| Aspect | System Prompt | User Prompt |
|--------|---------------|------------|
| Set by | Developer | User |
| Purpose | Define behavior | Request action |
| Visible to User | No (usually) | Yes |
| Changes | Rarely | Every interaction |
| Example | "You are an expert Python developer" | "How do I use decorators?" |
| Affects | Overall tone, capabilities | Specific response |

### Practical Code Example

```javascript
const messages = [
  {
    role: "system",
    content: `You are an expert JavaScript instructor. 
    Explain concepts clearly with code examples. 
    Assume the student has basic programming knowledge.`
  },
  {
    role: "user",
    content: "What is closure in JavaScript?"
  }
];

const response = await openai.chat.completions.create({
  model: "gpt-4",
  messages: messages,
  temperature: 0.7
});
```

### System Prompt for Different Use Cases

#### Coding Assistant:

```
You are an expert senior developer.
Provide production-ready code.
Explain your logic.
Consider edge cases.
Include error handling.
```

#### Content Writer:

```
You are a professional content writer with 10 years of experience.
Write engaging, SEO-optimized content.
Use an conversational tone.
Include relevant examples.
Keep paragraphs short (2-3 sentences max).
```

#### Interview Interviewer:

```
You are an experienced technical interviewer.
Ask progressive JavaScript questions.
Start with basics, get harder.
Provide thoughtful feedback.
Mark correct answers clearly.
Move to next topic after 2 correct answers.
```

### Why System Prompts Matter

1. **Consistency**: Same behavior across conversations
2. **Control**: Developer controls the model's personality
3. **Safety**: Can enforce rules and guidelines
4. **Efficiency**: Reduces repeated instructions
5. **Scalability**: Easy to apply to thousands of interactions

### Interview Answer

**Q: What's the difference between system and user prompts?**

**A:** A system prompt sets the model's behavior and personality before the user interaction. It's set by developers and defines rules, tone, and capabilities. A user prompt is the actual question/request from the user. Together, they guide the LLM's response. The system prompt is "be a helpful Python expert" while the user prompt is "help me with decorators."

## Topic 8: Hallucination (Critical Issue)

### What is Hallucination?

When an LLM **generates false, made-up information confidently**, as if it's true.

### Real Examples

#### Example 1: Fake Facts

```
User: "Tell me about the movie 'Quantum Bubbles' from 2015"

ChatGPT (Hallucinating):
"Quantum Bubbles is a 2015 sci-fi thriller directed by Christopher Nolan.
It stars Tom Hardy and features groundbreaking visual effects.
The film won 3 Academy Awards including Best Cinematography."

Reality: This movie doesn't exist. ChatGPT completely made it up.
```

#### Example 2: Fake Citations

```
User: "What does the research paper 'Advanced Neural Networks' by Dr. Smith say?"

ChatGPT (Hallucinating):
"According to 'Advanced Neural Networks' (Smith, 2019), 
neural networks can achieve 99.9% accuracy on all tasks."

Reality: The paper doesn't exist, or doesn't say that, or both.
```

#### Example 3: Confidently Wrong Information

```
User: "Who won the 2023 Nobel Prize in Physics?"

ChatGPT: "I don't have training data beyond April 2023, 
so I can't answer this reliably."

But if it did try to answer:
"The 2023 Nobel Prize in Physics was awarded to Dr. John Smith 
for his work on quantum computing."

Reality: Could be completely wrong (hallucinated).
```

### Why Does Hallucination Happen?

**Root Cause: LLMs are predictive, not knowledgeable**

```
LLM sees: "The inventor of the light bulb is..."
LLM thinks: "Based on my training, the most likely next tokens are 
[Thomas, Edison]. Let me predict that."

LLM doesn't think: "Let me check a database to verify this fact."
```

The model:
- ✅ Predicts plausible text
- ✅ Recognizes patterns
- ❌ Does NOT verify facts
- ❌ Does NOT access external databases
- ❌ Does NOT know when it's wrong

### Real-World Impact

**In Production, Hallucination Causes:**

```
Medical AI: Recommends non-existent treatments
Legal AI: Cites fake court cases
Financial AI: Gives false market data
Customer Service: Makes up return policies
```

### How to Detect Hallucination

**Red Flags:**

1. **Specific but unverifiable claims**
```
"Dr. Alan Turing published a paper in 1936..."
(Seems specific, but did he really?)
```

2. **Invented names/companies**
```
"According to TechCorp Inc., AI will..."
(Is TechCorp real? Who knows?)
```

3. **Confident tone with no source**
```
"The capital of France is Paris, according to recent research..."
(Why cite "research" for a basic fact?)
```

4. **Too detailed to be true**
```
"In 2019, AI research showed a 87.3% improvement in..."
(Suspiciously specific)
```

### Strategies to Prevent Hallucination

#### 1. RAG (Retrieval-Augmented Generation)

Provide the model with documents to reference:

```
Document: "The Eiffel Tower was built in 1889 by Gustave Eiffel."

User: "When was the Eiffel Tower built?"

LLM now has SOURCE MATERIAL to reference,
not just predictions.

Response: "The Eiffel Tower was built in 1889."

This is much more reliable.
```

#### 2. Prompt Guardrails

Tell the model to be honest about uncertainty:

```
System Prompt:
"If you're not sure about a fact, say 'I don't know' rather than guessing.
Cite sources when possible.
Flag any assumptions you make."
```

#### 3. Lower Temperature

Reduce randomness:

```
temperature = 0 (more deterministic, less hallucination)
vs
temperature = 1.5 (more creative, more hallucination)
```

#### 4. Fact-Checking Layer

Have a separate system verify facts:

```
LLM generates response
    ↓
Fact-checking system reviews claims
    ↓
If unverified, flag or regenerate
    ↓
Return response to user
```

#### 5. Prompt Instructions

Explicit instructions about accuracy:

```
"Answer only based on the documents provided.
If information isn't in the documents, say 'The documents don't contain this information.'
Do not make assumptions."
```

### Interview Answer

**Q: What is hallucination in LLMs?**

**A:** Hallucination occurs when an LLM confidently generates false information. It happens because LLMs predict likely text based on patterns, not because they verify facts. They don't have access to external databases. Solutions include RAG (providing documents to reference), explicit instructions to admit uncertainty, fact-checking layers, and lower temperature settings.

## Topic 9: Why RAG Exists (The Problem it Solves)

### The Problem

```
User: "Tell me about my company's latest product roadmap."

LLM Response: "I don't have information about your company. 
I was trained on public internet data until April 2024."
```

**The Issue:**
- GPT doesn't know about YOUR private documents
- Private company data isn't in public training data
- Asking about proprietary information returns generic answers

### A Worse Problem

```
User: "Tell me about my company's product."

Hallucinating LLM Response:
"Your company builds mobile apps for fitness tracking.
Your latest feature is AI-powered coaching.
Your market size is $500 million."

Reality: Your company makes enterprise software for hospitals.
The LLM completely made this up.
```

### The Solution: Provide Context

**Instead of (Bad):**
```
┌─────────────────────┐
│ User Question       │
│ "Tell me about..."  │
└──────────┬──────────┘
           ↓
    ┌─────────────┐
    │    LLM      │
    │ (No context)│
    └──────┬──────┘
           ↓
    ┌─────────────┐
    │  Hallucinated
    │  Response   │
    └─────────────┘
```

**Do this instead (Good):**
```
┌─────────────────────┐
│ Company Documents   │ ← Provide relevant documents
│ (10,000 tokens)     │
│ "Our product is..." │
└──────────┬──────────┘
           ↓
┌─────────────────────┐
│ User Question       │
│ "Tell me about..."  │
└──────────┬──────────┘
           ↓
    ┌─────────────┐
    │    LLM      │
    │ (Has context)
    └──────┬──────┘
           ↓
    ┌─────────────────────┐
    │ Informed Response   │
    │ (Based on docs)     │
    └─────────────────────┘
```

### Real Example

### Interview Answer

**Q: Why does RAG exist?**

**A:** RAG (Retrieval-Augmented Generation) solves the problem that LLMs don't have access to private company data and will hallucinate when asked about it. RAG works by:

## Topic 10: API-Based Architecture (Practical)

### The Architecture

```
┌──────────────┐
│  React UI    │  (User Interface)
└──────┬───────┘
       │ (HTTP Request)
       ↓
┌──────────────────┐
│  Node.js Backend │  (Your Server)
└──────┬───────────┘
       │ (API Call)
       ↓
┌─────────────────────┐
│  OpenAI/Claude API  │  (LLM Provider)
└──────┬──────────────┘
       │ (Response)
       ↓
┌──────────────┐
│  Node.js     │  (Process Response)
│  Backend     │
└──────┬───────┘
       │ (JSON Response)
       ↓
┌──────────────┐
│  React UI    │  (Display to User)
└──────────────┘
```

### Why This Architecture?

#### Direct Frontend → API (Bad)

```
Browser
  ↓
OpenAI API
  ↓
Browser

Problems:
- API key exposed in browser (security risk)
- CORS issues
- No control over requests
- Can't add authentication
- Rate limiting applies directly to users
```

#### Frontend → Backend → API (Good)

```
Browser
  ↓
Your Node.js Server (you control this)
  ↓
OpenAI API
  ↓
Your Node.js Server (you control this)
  ↓
Browser

Benefits:
- API key hidden on server
- You control the requests
- Add authentication/authorization
- Rate limiting per user, not global
- Can modify requests/responses
- Can log/monitor usage
- Can add custom logic
```

### Real Example: Chat Application

#### Frontend (React):

```javascript
import { useState } from 'react';

export function ChatApp() {
  const [message, setMessage] = useState('');
  const [responses, setResponses] = useState([]);

  const handleSend = async () => {
    // Send to YOUR backend, not OpenAI API
    const response = await fetch('/api/chat', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ message })
    });

    const data = await response.json();
    setResponses([...responses, data.reply]);
  };

  return (
    <div>
      <div>
        {responses.map(r => <p>{r}</p>)}
      </div>
      <input 
        value={message}
        onChange={(e) => setMessage(e.target.value)}
      />
      <button onClick={handleSend}>Send</button>
    </div>
  );
}
```

#### Backend (Node.js):

```javascript
import express from 'express';
import { Anthropic } from '@anthropic-ai/sdk';

const app = express();
app.use(express.json());

// Initialize Claude API client
const client = new Anthropic({
  apiKey: process.env.ANTHROPIC_API_KEY // Hidden on server
});

app.post('/api/chat', async (req, res) => {
  const { message } = req.body;

  // Add authentication
  if (!req.user) {
    return res.status(401).json({ error: 'Unauthorized' });
  }

  // Add rate limiting
  if (isRateLimited(req.user.id)) {
    return res.status(429).json({ error: 'Rate limited' });
  }

  // Call Claude API
  const response = await client.messages.create({
    model: 'claude-3-5-sonnet-20241022',
    max_tokens: 1024,
    system: 'You are a helpful assistant.',
    messages: [
      { role: 'user', content: message }
    ]
  });

  // Log for monitoring
  logUserQuery(req.user.id, message, response);

  // Send response back to frontend
  res.json({
    reply: response.content[0].text
  });
});

app.listen(3000);
```

### Key Points

#### 1. Frontend Never Calls API Directly

```
❌ WRONG:
fetch('https://api.anthropic.com/v1/messages', {
  headers: { 'Authorization': 'Bearer sk-...' } // Exposed!
})

✅ RIGHT:
fetch('http://localhost:3000/api/chat', {
  body: { message: 'Hello' }
})
```

#### 2. Backend Handles Authentication

```
Your backend:
- Verifies user is logged in
- Checks rate limits
- Validates requests
- Logs usage for billing
```

#### 3. API Key Hidden

```
Environment Variable (Server Only):
ANTHROPIC_API_KEY=sk-...

Browser never sees this.
```

#### 4. You Control the Flow

```
You can:
- Add custom validation
- Modify the prompt before sending
- Process the response
- Store conversation history in database
- Apply business logic
- Implement permission checks
```

### Full Chat Application Example

```javascript
import express from 'express';
import { Anthropic } from '@anthropic-ai/sdk';

const app = express();
app.use(express.json());

const client = new Anthropic({
  apiKey: process.env.ANTHROPIC_API_KEY
});

// In-memory conversation storage (use database in production)
const conversations = {};

app.post('/api/chat', async (req, res) => {
  const { userId, message } = req.body;

  // Initialize conversation for user if needed
  if (!conversations[userId]) {
    conversations[userId] = [];
  }

  // Add user message to history
  conversations[userId].push({
    role: 'user',
    content: message
  });

  try {
    // Call Claude with conversation history
    const response = await client.messages.create({
      model: 'claude-3-5-sonnet-20241022',
      max_tokens: 1024,
      system: 'You are a helpful customer service assistant.',
      messages: conversations[userId] // Include full history
    });

    const assistantMessage = response.content[0].text;

    // Add assistant response to history
    conversations[userId].push({
      role: 'assistant',
      content: assistantMessage
    });

    // Send response
    res.json({
      message: assistantMessage,
      tokenUsage: {
        input: response.usage.input_tokens,
        output: response.usage.output_tokens
      }
    });

  } catch (error) {
    console.error('API Error:', error);
    res.status(500).json({ error: 'Failed to process request' });
  }
});

app.listen(3000, () => {
  console.log('Chat server running on port 3000');
});
```

### Interview Answer

**Q: Describe a typical API architecture for an AI application.**

**A:** The standard architecture is: React Frontend → Node.js Backend → LLM API Provider.

The frontend sends user input to your Node.js backend (never directly to the LLM API). Your backend:
1. Authenticates the user
2. Applies rate limiting
3. Calls the LLM API with your API key (hidden from browser)
4. Processes the response
5. Returns it to the frontend

This approach keeps your API key secure, lets you control requests, add custom logic, and manage user limits. The backend acts as a secure intermediary between the client and the LLM provider.

### Practice Answers (Without Notes)

<details>
<summary>1. What is Generative AI?</summary>

Generative AI creates new content based on patterns.
Unlike traditional software that executes fixed logic,
generative AI predicts what should come next.
It creates text, code, images, and more.

</details>

<details>
<summary>2. What is an LLM?</summary>

An LLM (Large Language Model) is a neural network trained on 
massive amounts of text. It doesn't store facts in a database.
Instead, it predicts the statistically most likely next token
based on patterns it learned during training.

</details>

<details>
<summary>3. What are tokens?</summary>

Tokens are the smallest units LLMs understand - not words, not characters.
A word might be 1-2 tokens. They matter because:
- Context windows are measured in tokens
- API costs are measured in tokens
- They determine how much information fits

</details>

<details>
<summary>4. What is a context window?</summary>

Context window is the model's working memory - the maximum number
of tokens it can process in one request. For example, GPT-4 has
128k token context. Everything (prompt + docs + conversation)
must fit within this limit.

</details>

<details>
<summary>5. What does temperature control?</summary>

Temperature controls randomness in outputs.
At 0, the model is deterministic - always picks the most likely token.
At 1+, it becomes more creative and unpredictable.
Use temperature 0 for coding/facts, higher for creative writing.

</details>

<details>
<summary>6. What is prompt engineering?</summary>

Prompt engineering is writing better prompts to get better outputs.
Techniques include zero-shot (ask directly), few-shot (provide examples),
chain-of-thought (ask for step-by-step reasoning), and role-based
(give the model a persona).

</details>

<details>
<summary>7. Difference between system and user prompt?</summary>

System prompt sets the model's behavior/personality (set by developer).
User prompt is the actual question (from the user).
System: "You are a Python expert"
User: "How do I use decorators?"

</details>

<details>
<summary>8. What is hallucination?</summary>

Hallucination is when an LLM confidently generates false information.
It happens because LLMs predict likely text, not verify facts.
They don't access databases or check information. This is why
hallucination is common and dangerous in production.

</details>

<details>
<summary>9. Why does RAG exist?</summary>

RAG (Retrieval-Augmented Generation) solves hallucination on private data.
Instead of asking the LLM directly, you:
1. Retrieve relevant documents from your database
2. Provide them as context to the LLM
3. LLM answers based on the documents, not predictions

This makes answers accurate and grounded in real data.

</details>

<details>
<summary>10. Describe an API architecture</summary>

Frontend sends requests to your Node.js backend (not directly to LLM API).
Your backend authenticates, checks rate limits, calls the LLM with your
API key (hidden from browser), processes the response, and returns it.
This approach keeps your API key secure and lets you control requests.

</details>
