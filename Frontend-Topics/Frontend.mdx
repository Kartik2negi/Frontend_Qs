## 1. JavaScript Core

### 1.1 Closures

**What is a Closure?**

A closure is a function that has access to variables from another function's scope. This is possible because functions in JavaScript create closures around the data they need to work with.

```javascript
function outer(x) {
  function inner(y) {
    return x + y; // inner function accesses x from outer function
  }
  return inner;
}

const add5 = outer(5);
console.log(add5(3)); // 8
// The inner function 'remembers' x=5 even after outer() has finished executing
```

**Real-world Use Cases:**

#### 1. **Data Encapsulation (Private Variables)**
```javascript
function createCounter() {
  let count = 0; // private variable
  return {
    increment() { return ++count; },
    decrement() { return --count; },
    getCount() { return count; }
  };
}

const counter = createCounter();
counter.increment(); // 1
counter.increment(); // 2
console.log(counter.getCount()); // 2
// count cannot be accessed directly - it's private
```

#### 2. **Function Factory Pattern**
```javascript
function createMultiplier(multiplier) {
  return function(number) {
    return number * multiplier;
  };
}

const double = createMultiplier(2);
const triple = createMultiplier(3);
console.log(double(5)); // 10
console.log(triple(5)); // 15
```

#### 3. **Module Pattern**
```javascript
const module = (function() {
  const privateVar = "secret";
  
  return {
    publicMethod() {
      return privateVar;
    }
  };
})();
```

#### 4. **Callbacks and Event Handlers**
```javascript
function setupButton(buttonId, userId) {
  const button = document.getElementById(buttonId);
  button.addEventListener('click', function() {
    // This callback has access to userId through closure
    fetch(`/api/user/${userId}`).then(res => res.json());
  });
}
```

**Closure Memory Leaks:**

Closures can cause memory leaks if they hold references to large objects or the DOM when they're no longer needed.

```javascript
function problematicClosure() {
  const largeData = new Array(1000000).fill('data'); // Large object
  
  return function() {
    console.log(largeData[0]); // Even if we only use one element,
                                // the entire largeData stays in memory
  };
}

const fn = problematicClosure();
// largeData cannot be garbage collected while fn exists
```

**Solution: Clear references**
```javascript
function betterClosure() {
  let largeData = new Array(1000000).fill('data');
  
  return function() {
    console.log(largeData[0]);
    largeData = null; // Clean up when done
  };
}
```

### 1.2 Event Loop

The Event Loop is JavaScript's mechanism for handling asynchronous operations in a single-threaded environment.

**Output Example - What will be printed?**

```javascript
console.log(1);              // Call Stack → Output: 1

setTimeout(() => console.log(2)); // Goes to Web API, then Macrotask Queue

Promise.resolve().then(() => console.log(3)); // Goes to Microtask Queue

console.log(4);              // Call Stack → Output: 4

// Expected Output:
// 1
// 4
// 3
// 2
```

**Explanation:**

1. **Synchronous code** executes first (1, then 4)
2. **Microtasks** execute next (Promise - 3)
3. **Macrotasks** execute last (setTimeout - 2)

**Detailed Flow:**

```
┌─────────────────────────────────────────────────────┐
│            JavaScript Runtime                        │
│  ┌──────────────┐  ┌──────────────┐  ┌───────────┐  │
│  │  Call Stack  │  │ Microtask Q  │  │Macrotask Q│  │
│  └──────────────┘  └──────────────┘  └───────────┘  │
└─────────────────────────────────────────────────────┘
         ↓
   Event Loop continuously:
   1. Execute all sync code (Call Stack)
   2. Execute all microtasks (empty microtask queue)
   3. Execute ONE macrotask
   4. Repeat from step 2
```

**More Complex Example:**
```javascript
console.log('Start');

setTimeout(() => {
  console.log('setTimeout 1');
  Promise.resolve().then(() => console.log('Promise in setTimeout'));
}, 0);

Promise.resolve()
  .then(() => {
    console.log('Promise 1');
    setTimeout(() => console.log('setTimeout in Promise'), 0);
  })
  .then(() => console.log('Promise 2'));

console.log('End');

// Output:
// Start
// End
// Promise 1
// Promise 2
// setTimeout 1
// Promise in setTimeout
// setTimeout in Promise
```

### 1.3 Microtask vs Macrotask

**Microtasks** execute immediately after the current execution context, before rendering.

**Macrotasks** execute after rendering.

```javascript
// MICROTASKS (High Priority)
- Promise.resolve().then()
- queueMicrotask()
- MutationObserver
- process.nextTick() (Node.js)

// MACROTASKS (Lower Priority)
- setTimeout()
- setInterval()
- setImmediate() (Node.js)
- requestAnimationFrame()
- I/O operations
- UI rendering
```

**Priority Example:**
```javascript
console.log('1: Sync');

setTimeout(() => console.log('2: setTimeout'), 0);

Promise.resolve().then(() => console.log('3: Promise'));

queueMicrotask(() => console.log('4: queueMicrotask'));

console.log('5: Sync');

// Output:
// 1: Sync
// 5: Sync
// 3: Promise
// 4: queueMicrotask
// 2: setTimeout
```

### 1.4 Hoisting

Hoisting is JavaScript's behavior of moving declarations to the top of their scope.

**var Hoisting:**
```javascript
console.log(a); // undefined (not ReferenceError)
var a = 10;
console.log(a); // 10

// Interpreted as:
// var a;
// console.log(a); // undefined
// a = 10;
// console.log(a); // 10
```

**Function Hoisting:**
```javascript
sayHi(); // "Hello" - works fine

function sayHi() {
  console.log('Hello');
}

// Function declarations are fully hoisted
```

**let and const do NOT hoist the same way:**
```javascript
console.log(b); // ReferenceError
let b = 10;
```

### 1.5 Temporal Dead Zone (TDZ)

Variables declared with `let` and `const` are hoisted but not initialized. They exist in a "Temporal Dead Zone" from the start of the block until the declaration.

```javascript
console.log(x); // ReferenceError: Cannot access 'x' before initialization
let x = 10;

// This is the TDZ ↑
```

**TDZ with Function Parameters:**
```javascript
function test(a = b, b) {
  // b is still in TDZ when default value of a is evaluated
}
test(undefined, 1); // ReferenceError
```

### 1.6 Call, Apply, Bind

These methods are used to control what `this` refers to.

**call()** - Invokes function immediately with specified `this`
```javascript
function greet(greeting) {
  return `${greeting}, ${this.name}`;
}

const person = { name: 'John' };
console.log(greet.call(person, 'Hello')); // Hello, John
```

**apply()** - Same as call but takes arguments as array
```javascript
greet.apply(person, ['Hi']); // Hi, John
```

**bind()** - Returns new function with bound `this`
```javascript
const boundGreet = greet.bind(person);
boundGreet('Hey'); // Hey, John
```

**Implement Custom bind():**
```javascript
Function.prototype.customBind = function(context, ...args) {
  const originalFunc = this;
  
  return function(...laterArgs) {
    return originalFunc.apply(context, [...args, ...laterArgs]);
  };
};

// Usage:
const boundGreet = greet.customBind(person, 'Hello');
console.log(boundGreet()); // Hello, John
```

### 1.7 Debounce

Debounce delays function execution until after a specified time has passed since the last invocation.

```javascript
function debounce(func, delay) {
  let timeoutId;
  
  return function(...args) {
    clearTimeout(timeoutId);
    timeoutId = setTimeout(() => {
      func.apply(this, args);
    }, delay);
  };
}

// Real-world example: Search API calls
const searchAPI = (query) => console.log(`Searching for: ${query}`);
const debouncedSearch = debounce(searchAPI, 500);

// Simulating user typing
debouncedSearch('r');    // cleared
debouncedSearch('re');   // cleared
debouncedSearch('react'); // This executes after 500ms
```

### 1.8 Throttle

Throttle ensures function executes at most once every specified time interval.

```javascript
function throttle(func, limit) {
  let inThrottle;
  
  return function(...args) {
    if (!inThrottle) {
      func.apply(this, args);
      inThrottle = true;
      
      setTimeout(() => {
        inThrottle = false;
      }, limit);
    }
  };
}

// Real-world example: Scroll event
const handleScroll = () => console.log('Scrolling');
const throttledScroll = throttle(handleScroll, 1000);

// Called multiple times but executes max once per second
window.addEventListener('scroll', throttledScroll);
```

**Debounce vs Throttle:**
- **Debounce**: Wait until user stops for X ms
- **Throttle**: Execute at most once every X ms

### 1.9 Currying

Currying transforms a function with multiple arguments into a chain of functions with single arguments.

```javascript
// Regular function
function sum(a, b, c) {
  return a + b + c;
}

// Curried version
function curriedSum(a) {
  return function(b) {
    return function(c) {
      return a + b + c;
    };
  };
}

console.log(curriedSum(1)(2)(3)); // 6

// Arrow function version
const curriedAdd = a => b => c => a + b + c;
console.log(curriedAdd(1)(2)(3)); // 6
```

**Generic Currying Function:**
```javascript
function curry(fn) {
  return function curried(...args) {
    if (args.length >= fn.length) {
      return fn.apply(this, args);
    } else {
      return function(...nextArgs) {
        return curried.apply(this, args.concat(nextArgs));
      };
    }
  };
}

const add = (a, b, c) => a + b + c;
const curriedAdd = curry(add);

console.log(curriedAdd(1)(2)(3)); // 6
console.log(curriedAdd(1)(2, 3)); // 6
console.log(curriedAdd(1, 2)(3)); // 6
```

### 1.10 Deep Copy vs Shallow Copy

**Shallow Copy:** Copies only the top level. Nested objects are still referenced.

```javascript
const original = {
  name: 'John',
  address: { city: 'NYC' }
};

const shallow = { ...original };
shallow.name = 'Jane'; // original.name stays 'John'
shallow.address.city = 'LA'; // original.address.city becomes 'LA' (problem!)
```

**Deep Copy:** Copies everything recursively.

```javascript
// Method 1: structuredClone (recommended - modern browsers)
const deep = structuredClone(original);
deep.address.city = 'LA'; // original stays unchanged

// Method 2: JSON (doesn't work with functions, dates, undefined)
const deep2 = JSON.parse(JSON.stringify(original));

// Method 3: Recursive function
function deepCopy(obj) {
  if (obj === null || typeof obj !== 'object') return obj;
  
  const copy = Array.isArray(obj) ? [] : {};
  
  for (let key in obj) {
    if (obj.hasOwnProperty(key)) {
      copy[key] = deepCopy(obj[key]);
    }
  }
  
  return copy;
}
```

### 1.11 Memory Leaks in JavaScript

**Common Causes:**

#### 1. **Event Listeners Not Removed**
```javascript
function setup() {
  const element = document.getElementById('btn');
  element.addEventListener('click', handler);
  // If element is removed but listener isn't, memory leaks
}

// Fix:
function setup() {
  const element = document.getElementById('btn');
  element.addEventListener('click', handler);
  
  return function cleanup() {
    element.removeEventListener('click', handler);
  };
}
```

#### 2. **Timers Not Cleared**
```javascript
// Bad
setInterval(() => doSomething(), 1000); // Never stops

// Good
const intervalId = setInterval(() => doSomething(), 1000);
clearInterval(intervalId); // Clean up when done
```

#### 3. **Closures Holding Large Objects**
```javascript
function createHandler() {
  const largeData = new Array(1000000).fill('data');
  
  return function handler() {
    console.log(largeData[0]); // largeData persists in memory
  };
}
```

#### 4. **Detached DOM Nodes**
```javascript
// Bad
let detachedNode = document.getElementById('node');
document.body.removeChild(detachedNode);
// detachedNode variable still references the removed node

// Good
let node = document.getElementById('node');
document.body.removeChild(node);
node = null; // Release reference
```

#### 5. **Global Variables**
```javascript
// Bad
window.cache = []; // Global cache grows indefinitely

// Good
const cache = []; // Scoped variable
// Add cleanup logic
```

### 1.12 Async/Await

Modern way to handle asynchronous code.

```javascript
// Basic usage
async function fetchUser(id) {
  try {
    const response = await fetch(`/api/user/${id}`);
    const data = await response.json();
    return data;
  } catch (error) {
    console.error('Failed to fetch:', error);
  }
}
```

**Promise APIs:**

```javascript
// Promise.all - waits for all, fails if any rejects
const results = await Promise.all([
  fetch('/api/user'),
  fetch('/api/posts'),
  fetch('/api/comments')
]);

// Promise.allSettled - waits for all, returns their status
const results = await Promise.allSettled([
  fetch('/api/user'),
  fetch('/api/posts')
]);
// Returns: [{ status: 'fulfilled', value: ... }, { status: 'rejected', reason: ... }]

// Promise.race - returns result of first settled promise
const result = await Promise.race([
  fetch('/api/fast'),
  fetch('/api/slow')
]);

// Promise.any - returns result of first fulfilled promise
const result = await Promise.any([
  fetch('/mirror1'),
  fetch('/mirror2'),
  fetch('/mirror3')
]);
```

**Error Handling:**
```javascript
async function handleErrors() {
  try {
    const data = await fetch('/api/data').then(r => r.json());
    return data;
  } catch (error) {
    // Handle fetch error
    console.error(error);
    throw error;
  } finally {
    // Cleanup code
  }
}
```

## 2. Browser Internals

### 2.1 Critical Rendering Path (CRP)

The sequence of steps the browser takes to convert HTML, CSS, and JavaScript into rendered pixels.

```
1. PARSING
   ↓
   HTML → DOM Tree
   CSS → CSSOM Tree
   
2. RENDERING
   ↓
   DOM + CSSOM → Render Tree
   
3. LAYOUT (Reflow)
   ↓
   Calculate positions and dimensions
   
4. PAINT (Repaint)
   ↓
   Draw pixels to layers
   
5. COMPOSITE
   ↓
   Combine layers into final image
```

**Timeline:**
```
HTML parsing begins
  ↓
CSS blocks rendering (must parse all CSS)
  ↓
JavaScript blocks parsing (synchronous scripts)
  ↓
DOM is ready
  ↓
Render Tree is constructed
  ↓
Layout calculations
  ↓
Paint to layers
  ↓
Composite layers
  ↓
Page visible
```

**Optimization:**
```html
<!-- CSS in head - blocks rendering but necessary -->
<head>
  <link rel="stylesheet" href="style.css">
</head>

<body>
  <h1>Content</h1>
  
  <!-- JavaScript at end - doesn't block parsing -->
  <script src="script.js"></script>
  
  <!-- Or use async/defer -->
  <script async src="script.js"></script>
  <script defer src="script.js"></script>
</body>
```

### 2.2 Reflow vs Repaint

**Repaint:**
- Triggered when visual properties change (color, background, visibility)
- Less expensive
- Doesn't affect layout

```javascript
element.style.color = 'red'; // Repaint
element.style.visibility = 'hidden'; // Repaint
```

**Reflow:**
- Triggered when layout properties change
- Expensive (recalculates entire layout)
- Causes repaint as well

```javascript
element.style.width = '200px'; // Reflow + Repaint
element.style.height = '100px'; // Reflow + Repaint
element.style.margin = '10px'; // Reflow + Repaint
```

**Minimize Reflows:**
```javascript
// Bad - multiple reflows
element.style.width = '100px';
element.style.height = '100px';
element.style.margin = '10px';

// Good - single reflow
element.style.cssText = 'width: 100px; height: 100px; margin: 10px;';

// Also good - batch changes
const style = element.style;
style.width = '100px';
style.height = '100px';
style.margin = '10px';

// Even better - use classes
element.classList.add('styled');
```

**Forced Reflow:**
```javascript
// Bad - forces reflow in loop
for (let i = 0; i < 1000; i++) {
  elements[i].style.width = elements[i].offsetWidth + 10 + 'px';
  // Reading offsetWidth forces layout recalculation
}

// Good - read all, then write all
const newWidths = elements.map(el => el.offsetWidth + 10);
elements.forEach((el, i) => {
  el.style.width = newWidths[i] + 'px';
});
```

### 2.3 What Blocks Rendering?

**CSS:**
- CSS blocks rendering
- Browser won't display anything until CSS is parsed
- This is intentional (prevent flash of unstyled content)

```html
<head>
  <link rel="stylesheet" href="style.css"> <!-- Blocks rendering -->
</head>
```

**Synchronous JavaScript:**
- Synchronous `<script>` tags block HTML parsing
- Browser stops parsing to execute the script

```html
<script src="app.js"></script> <!-- Blocks parsing -->

<!-- No HTML below this is parsed until app.js executes -->
<div>This might not exist when app.js runs</div>
```

**Solutions:**
```html
<!-- Deferred execution -->
<script defer src="app.js"></script> <!-- Doesn't block parsing -->

<!-- Asynchronous execution -->
<script async src="analytics.js"></script> <!-- Doesn't block parsing -->

<!-- Execute after parsing -->
<script>
  document.addEventListener('DOMContentLoaded', () => {
    // DOM is ready
  });
</script>
```

### 2.4 Browser Storage

## 3. React Core

### 3.1 React Fiber

**Why Introduced?**

React Fiber was introduced to solve the problem of long-running JavaScript blocking the main thread. The old reconciliation algorithm was synchronous and couldn't be interrupted.

**Problems with Old System:**
```javascript
// Old: Synchronous rendering
// If rendering takes 20ms and there's a user input, 
// the input waits 20ms - janky experience
```

**Fiber Solution:**
- **Pausing**: Can pause rendering work
- **Prioritization**: Can prioritize important work (user input)
- **Resume**: Can resume work later
- **Incremental**: Can split work into chunks

**How Fiber Works:**
```
┌─────────────────────┐
│  Reconciliation     │
│  (can be paused)    │
│  ┌───────────────┐  │
│  │ Fiber 1 (5ms) │  │
│  │ Fiber 2 (5ms) │  │
│  │ Fiber 3 (5ms) │  │
│  └───────────────┘  │
│  ↓ (User input?)    │
│  ↓ Pause & resume   │
└─────────────────────┘
         ↓
┌─────────────────────┐
│  Commit            │
│  (synchronous)     │
│  Apply changes     │
└─────────────────────┘
```

**Cooperative Scheduling:**
```javascript
// React doesn't block the entire render
// It yields to browser for:
// - User input
// - Higher priority updates
// - requestAnimationFrame

// Example: Low priority update (state change from event)
flushSync(() => {
  setCount(c => c + 1); // High priority
});

startTransition(() => {
  setFilteredList(filter(items)); // Low priority, can be interrupted
});
```

### 3.2 Reconciliation

Reconciliation is React's algorithm to determine what DOM changes are necessary.

**Diffing Algorithm:**

```javascript
// Old tree
<ul>
  <li key="a">A</li>
  <li key="b">B</li>
  <li key="c">C</li>
</ul>

// New tree
<ul>
  <li key="a">A</li>
  <li key="c">C</li>
  <li key="d">D</li>
</ul>

// React diffs:
// - key="a" exists, update if needed
// - key="b" is gone, remove
// - key="c" exists, update if needed  
// - key="d" is new, add
```

**Key Rules:**
1. Elements of different types produce different trees
2. Keys help identify which items changed

```javascript
// BAD - don't use index as key if list can change
{items.map((item, index) => <li key={index}>{item}</li>)}

// GOOD - use unique identifier
{items.map(item => <li key={item.id}>{item}</li>)}
```

**Why Keys Matter:**
```javascript
// Without key (uses index)
const Component = () => {
  const [items, setItems] = useState(['A', 'B', 'C']);
  
  return (
    <ul>
      {items.map((item, i) => (
        <li key={i}>
          <input placeholder={item} />
        </li>
      ))}
    </ul>
  );
};

// If you remove 'A', the list becomes ['B', 'C']
// But input values stay with indices, causing mismatch

// With proper key
{items.map(item => (
  <li key={item}>
    <input placeholder={item} />
  </li>
))}
// Now input values move with items
```

### 3.3 Virtual DOM

The Virtual DOM is React's in-memory representation of the actual DOM.

**Why Needed:**

1. **Performance**: Batch updates instead of updating DOM immediately
2. **Abstraction**: Developers work with React, not directly with DOM
3. **Optimization**: React can calculate minimal changes needed

```javascript
// Without React (direct DOM manipulation)
document.getElementById('count').textContent = '0';
document.getElementById('count').textContent = '1';
document.getElementById('count').textContent = '2';
// 3 DOM updates

// With React
const [count, setCount] = useState(0);
// Batches multiple setState calls
setCount(1);
setCount(1); // deduplicated
setCount(2); // batched, only updates DOM once
```

**Virtual DOM Process:**

```
┌────────────────────┐
│  setState() called  │
└────────────────────┘
         ↓
┌────────────────────┐
│ Create new Vdom    │
└────────────────────┘
         ↓
┌────────────────────┐
│ Diff with old Vdom │
└────────────────────┘
         ↓
┌────────────────────┐
│ Calculate patches  │
└────────────────────┘
         ↓
┌────────────────────┐
│ Apply to real DOM  │
└────────────────────┘
```

### 3.4 Rendering Lifecycle

**Parent Rerender → Child Rerender Pattern:**

```javascript
function Parent() {
  const [count, setCount] = useState(0);
  
  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>Click</button>
      <Child count={count} /> {/* Rerenders when count changes */}
    </div>
  );
}

function Child({ count }) {
  console.log('Child renders'); // Logs every time parent updates
  return <div>Count: {count}</div>;
}
```

**Why This Happens:**

React re-renders parent, and by default, also re-renders all children. This is safe but can be inefficient.

**When Does It Happen:**

```javascript
// Child rerenders when:
// 1. Parent rerenders AND
// 2. Child component function is called

function Parent() {
  const [count, setCount] = useState(0);
  const [name, setName] = useState('John');
  
  return (
    <>
      {/* Every state change triggers Parent rerender */}
      {/* And subsequently calls Child function */}
      <Child count={count} name={name} />
    </>
  );
}
```

**Preventing Unnecessary Rerenders:**

```javascript
// Option 1: React.memo
const Child = React.memo(function Child({ count }) {
  return <div>Count: {count}</div>;
});

// Option 2: useCallback for stable references
function Parent() {
  const [count, setCount] = useState(0);
  
  const handleClick = useCallback(() => {
    // Only recreated if dependencies change
  }, []);
  
  return <Child onClick={handleClick} />;
}

// Option 3: useMemo
const memoChild = useMemo(() => <Child />, [dependencies]);
```

### 3.5 React.memo

`React.memo` prevents rerender if props haven't changed.

```javascript
const Child = React.memo(function Child({ name }) {
  console.log('Child renders');
  return <div>Hello {name}</div>;
});

function Parent() {
  const [count, setCount] = useState(0);
  const [name, setName] = useState('John');
  
  return (
    <>
      <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>
      <Child name={name} /> {/* Only rerenders if name changes */}
    </>
  );
}
```

**When to Use:**

```javascript
// Good use case: Expensive component
const ExpensiveList = React.memo(({ items }) => {
  const sorted = items.sort((a, b) => a - b); // Expensive operation
  return items.map(item => <div key={item}>{item}</div>);
});

// Good use case: Pure component (same props = same output)
const Timestamp = React.memo(({ date }) => {
  return <div>{date.toLocaleString()}</div>;
});
```

**When NOT to Use:**

```javascript
// Bad: Props are always different (functions)
const BadMemo = React.memo(({ onClick }) => {
  return <button onClick={onClick}>Click</button>;
});

function Parent() {
  return (
    <BadMemo onClick={() => console.log('clicked')} />
    // onClick is new every render, memo is useless
  );
}

// Bad: Props are always different (objects)
const BadMemo2 = React.memo(({ style }) => {
  return <div style={style}>Text</div>;
});

function Parent() {
  return (
    <BadMemo2 style={{ color: 'red' }} />
    // style object is new every render
  );
}
```

### 3.6 useMemo

Memoizes expensive computations.

```javascript
const expensiveValue = useMemo(() => {
  return expensiveCalculation(data);
}, [data]); // Recalculate only if data changes
```

**When Useful:**

```javascript
// Expensive computation
function Component({ items }) {
  const sortedItems = useMemo(() => {
    return items.sort((a, b) => a.price - b.price);
  }, [items]);
  
  return (
    <div>
      {sortedItems.map(item => <Item key={item.id} item={item} />)}
    </div>
  );
}

// Preventing child rerenders (stable object)
function Parent({ data }) {
  const memoData = useMemo(() => ({ ...data }), [data]);
  
  return <Child data={memoData} />;
}
```

**When Harmful:**

```javascript
// Premature optimization - simple values don't need useMemo
const doubled = useMemo(() => count * 2, [count]); // Unnecessary

// Adding overhead instead of saving time
const simple = useMemo(() => count + 1, [count]); // Overhead > benefit

// Every dependency needs useMemo (unnecessary cascade)
const a = useMemo(() => calculation1(), []);
const b = useMemo(() => calculation2(a), [a]);
const c = useMemo(() => calculation3(b), [b]);
```

### 3.7 useCallback

Memoizes function references.

```javascript
const memoCallback = useCallback(() => {
  doSomething(a, b);
}, [a, b]); // Function identity only changes if a or b changes
```

**When Useful:**

```javascript
// Passing callbacks to memoized children
function Parent() {
  const [count, setCount] = useState(0);
  
  const handleClick = useCallback(() => {
    setCount(c => c + 1);
  }, []);
  
  return <MemoChild onClick={handleClick} />; // Stable reference
}

// Dependency for other hooks
function Form() {
  const [values, setValues] = useState({});
  
  const handleChange = useCallback((e) => {
    setValues(prev => ({ ...prev, [e.target.name]: e.target.value }));
  }, []);
  
  useEffect(() => {
    // This dependency is stable, effect doesn't run on every render
  }, [handleChange]);
}
```

**When Harmful:**

```javascript
// Every callback wrapped (unnecessary)
function Component() {
  const log = useCallback(() => console.log('test'), []);
  const add = useCallback((a, b) => a + b, []);
  // Creating closures is more expensive than inline functions
}

// Complex dependency arrays
const callback = useCallback(() => {
  heavyCalculation(a, b, c, d, e, f, g);
}, [a, b, c, d, e, f, g]); // Callback changes constantly
```

### 3.8 React.lazy & Suspense

**React.lazy:**

**How Suspense Works:**

```javascript
function Suspense({ fallback, children }) {
  try {
    return children; // Renders component
  } catch (promise) {
    // Component threw promise (still loading)
    // Show fallback until promise resolves
    return fallback;
  }
}
```

**Practical Example:**

```javascript
      {currentPage === 'settings' && <Settings />}
      {currentPage === 'profile' && <Profile />}
    </Suspense>
  );
}
```

### 3.9 Stale Closures

Stale closure occurs when a function captures old variable values.

**Example:**

```javascript
function Counter() {
  const [count, setCount] = useState(0);
  
  useEffect(() => {
    const timer = setInterval(() => {
      console.log(count); // Always logs 0 (stale)
    }, 1000);
    
    return () => clearInterval(timer);
  }, []); // Missing count dependency
  
  return <div>{count}</div>;
}

// count is 0 when effect runs
// count changes to 1, 2, 3... but effect doesn't rerun
// closure still has count = 0
```

**Solution:**

```javascript
// Option 1: Add to dependency array
useEffect(() => {
  const timer = setInterval(() => {
    console.log(count); // Always current
  }, 1000);
  
  return () => clearInterval(timer);
}, [count]); // Include count

// Option 2: Use functional update
useEffect(() => {
  const timer = setInterval(() => {
    setCount(prevCount => {
      console.log(prevCount); // Always current
      return prevCount;
    });
  }, 1000);
  
  return () => clearInterval(timer);
}, []); // Can be empty
```

### 3.10 Controlled vs Uncontrolled Components

**Controlled Component:**
React state is the "source of truth"

```javascript
function ControlledInput() {
  const [value, setValue] = useState('');
  
  return (
    <input
      value={value} // Controlled by state
      onChange={(e) => setValue(e.target.value)}
    />
  );
}
```

**Uncontrolled Component:**
DOM is the "source of truth"

```javascript
function UncontrolledInput() {
  const inputRef = useRef(null);
  
  const handleClick = () => {
    console.log(inputRef.current.value); // Get value from DOM
  };
  
  return (
    <>
      <input ref={inputRef} /> {/* Uncontrolled */}
      <button onClick={handleClick}>Get Value</button>
    </>
  );
}
```

**When to Use:**

| Controlled | Uncontrolled |
|-----------|-------------|
| Need instant validation | File inputs |
| Complex logic | One-off value reads |
| Conditional rendering | Simple forms |
| Integration with React | Legacy code |

### 4.6 Memoization Strategy

**Decision Tree:**

```
Is component slow?
├─ Expensive computation?
│  └─ useMemo
├─ Expensive child component?
│  └─ React.memo + useCallback
├─ Rerendering unnecessarily?
│  └─ Check dependencies
└─ All good, skip optimization
```

**Example Decision Process:**

```javascript
function ProductList({ products, filter }) {
  // Expensive filter operation
  const filteredProducts = useMemo(() => {
    return products.filter(p => p.name.includes(filter));
  }, [products, filter]);
  
  // Pass stable callback to memoized child
  const handleSelect = useCallback((id) => {
    console.log('Selected:', id);
  }, []);
  
  return (
    <ul>
      {filteredProducts.map(product => (
        <MemoProduct 
          key={product.id} 
          product={product}
          onSelect={handleSelect}
        />
      ))}
    </ul>
  );
}

const MemoProduct = React.memo(function Product({ product, onSelect }) {
  return (
    <li onClick={() => onSelect(product.id)}>
      {product.name}
    </li>
  );
});
```

## 5. State Management

### 5.1 Redux

**Core Concepts:**

```javascript
// Store: centralized state
const store = createStore(reducer);

// Reducer: pure function that updates state
function reducer(state = initialState, action) {
  switch(action.type) {
    case 'INCREMENT':
      return { ...state, count: state.count + 1 };
    default:
      return state;
  }
}

// Action: describes what happened
const action = { type: 'INCREMENT' };

// Dispatch: sends action to store
store.dispatch(action);

// Subscribe: listens to state changes
store.subscribe(() => {
  console.log(store.getState());
});
```

**Flow:**
```
User Input
    ↓
Dispatch Action
    ↓
Action → Reducer
    ↓
Reducer returns new state
    ↓
Store updates
    ↓
Components rerender
```

### 6.3 Hydration Mismatch

## 8. Security

### 8.1 XSS (Cross-Site Scripting)

Injecting malicious scripts into web pages.

```javascript
// ❌ Vulnerable
function Component() {
  const userInput = '<img src=x onerror="alert(\'hacked\')">';
  return <div dangerouslySetInnerHTML={{ __html: userInput }} />;
}

// ✅ Safe - React escapes by default
function Component() {
  const userInput = '<img src=x onerror="alert(\'hacked\')">';
  return <div>{userInput}</div>; // Rendered as text, not HTML
}
```

**Prevention:**

```javascript
// Always escape user input
function Component({ userComment }) {
  return <div>{userComment}</div>; // Safe

  // NOT:
  // return <div dangerouslySetInnerHTML={{ __html: userComment }} />;
}

// Sanitize if you must render HTML
import DOMPurify from 'dompurify';

function SafeHTML({ html }) {
  const clean = DOMPurify.sanitize(html);
  return <div dangerouslySetInnerHTML={{ __html: clean }} />;
}
```

### 8.2 CSRF (Cross-Site Request Forgery)

Attacker tricks user into performing unwanted actions.

```html
<!-- Attacker's page -->
<img src="https://yourbank.com/transfer?amount=1000&to=attacker" />
<!-- If user is logged in, request is sent with user's cookies -->
```

**Prevention:**

```javascript
// 1. Use CSRF tokens
<form method="POST" action="/api/transfer">
  <input type="hidden" name="csrf" value={csrfToken} />
  <input type="number" name="amount" />
  <button>Transfer</button>
</form>

// 2. Server validates token
app.post('/api/transfer', (req, res) => {
  if (req.body.csrf !== req.session.csrfToken) {
    return res.status(403).json({ error: 'Invalid token' });
  }
  // Process transfer
});

// 3. Use SameSite cookies
res.cookie('session', token, {
  sameSite: 'strict' // Only sent to same site
});
```

## 11. TypeScript

### 11.1 Interface vs Type

```typescript
// Interface - used for objects
interface User {
  name: string;
  email: string;
  greet(): void;
}

// Type - more flexible
type User = {
  name: string;
  email: string;
  greet(): void;

interface Admin extends User {
  role: 'admin';

type Admin = User & { role: 'admin' };

interface Window {
  myCustomProperty: string;

type Status = 'pending' | 'completed' | 'failed';

// Best practice:
// Use interface for object contracts
// Use type for unions, tuples, functions
```

### 11.6 Unknown vs Any

```typescript
// any - opt-out of type checking
let value: any = 'string';
value.trim(); // No error, assumes it exists
value = 123;
value.toFixed(); // No error, but actually it's a string

// unknown - unknown type (safer)
let unknownValue: unknown = 'string';
// unknownValue.trim(); // Error: unknown doesn't have trim

// Must narrow type first
if (typeof unknownValue === 'string') {
  unknownValue.trim(); // Now valid
}

// Type guard
function assertString(value: unknown): asserts value is string {
  if (typeof value !== 'string') {
    throw new Error('Not a string');
  }
}

assertString(unknownValue);
unknownValue.trim(); // Now guaranteed to be string

// Best practice: avoid both, use specific types
// If you can't know the type, use unknown
// Never use any unless absolutely necessary
```

```javascript
  const [query, setQuery] = useState('');
  const timerRef = useRef(null);
```

```javascript
function InfiniteScroll() {
  const [items, setItems] = useState([]);
  const [page, setPage] = useState(0);
  const [loading, setLoading] = useState(false);
  const observerTarget = useRef(null);
  
  useEffect(() => {
    const observer = new IntersectionObserver(entries => {
      if (entries[0].isIntersecting && !loading) {
        loadMore();
      }
    });
    
    if (observerTarget.current) {
      observer.observe(observerTarget.current);
    }
    
    return () => observer.disconnect();
  }, [loading]);

    setLoading(true);
    const res = await fetch(`/api/items?page=${page}`);
    const newItems = await res.json();
    setItems(prev => [...prev, ...newItems]);
    setPage(prev => prev + 1);
    setLoading(false);
  };
  
  useEffect(() => {
    loadMore();
  }, []); // Load initial items
  
  return (
    <>
      {items.map(item => <Item key={item.id} item={item} />)}
      <div ref={observerTarget}>{loading && 'Loading...'}</div>
    </>
  );
}
```

```javascript
function useDebounce(value, delay) {
  const [debouncedValue, setDebouncedValue] = useState(value);
  
  useEffect(() => {
    const handler = setTimeout(() => {
      setDebouncedValue(value);
    }, delay);
    
    return () => clearTimeout(handler);
  }, [value, delay]);
  
  return debouncedValue;
}
```

### 14.6 useThrottle Hook

```javascript
function useThrottle(value, delay) {
  const [throttledValue, setThrottledValue] = useState(value);
  const lastUpdated = useRef(Date.now());
  
  useEffect(() => {
    const now = Date.now();
    
    if (now >= lastUpdated.current + delay) {
      lastUpdated.current = now;
      setThrottledValue(value);
    } else {
      const timer = setTimeout(() => {
        lastUpdated.current = Date.now();
        setThrottledValue(value);
      }, delay - (now - lastUpdated.current));
      
      return () => clearTimeout(timer);
    }
  }, [value, delay]);
  
  return throttledValue;
}
```

### 14.7 useFetch Hook

```javascript
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);
  
  useEffect(() => {
    const abortController = new AbortController();
    
    const fetchData = async () => {
      try {
        const response = await fetch(url, {
          signal: abortController.signal
        });
        
        if (!response.ok) {
          throw new Error(`HTTP error! status: ${response.status}`);
        }
        
        const json = await response.json();
        setData(json);
        setError(null);
      } catch (err) {
        if (err.name !== 'AbortError') {
          setError(err.message);
        }
      } finally {
        setLoading(false);
      }
    };
    
    fetchData();
    
    return () => abortController.abort();
  }, [url]);
  
  return { data, loading, error };
}
```

### 16.1 Error Boundaries

Error Boundaries catch errors in React component tree and display fallback UI.

```typescript
// ErrorBoundary.tsx
import React from 'react';

interface Props {
  children: React.ReactNode;
  fallback?: React.ReactNode;
}

interface State {
  hasError: boolean;
  error: Error | null;
}

class ErrorBoundary extends React.Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false, error: null };
  }

  static getDerivedStateFromError(error: Error) {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, errorInfo: React.ErrorInfo) {
    // Log to error tracking service (Sentry, LogRocket)
    console.error('Error caught:', error, errorInfo);
    
    // Send to Sentry
    Sentry.captureException(error, { contexts: { errorInfo } });
  }

  render() {
    if (this.state.hasError) {
      return (
        <div style={{ padding: '20px', textAlign: 'center' }}>
          <h2>Something went wrong</h2>
          <p>{this.state.error?.message}</p>
          <button onClick={() => this.setState({ hasError: false })}>
            Try Again
          </button>
        </div>
      );
    }

    return this.props.children;
  }
}

// Usage
export default function App() {
  return (
    <ErrorBoundary fallback={<div>Error occurred</div>}>
      <YourComponent />
    </ErrorBoundary>
  );
}
```

**What Error Boundaries DON'T catch:**
- ❌ Async errors (use try-catch)
- ❌ Event handler errors (use try-catch)
- ❌ Server-side rendering
- ❌ Errors in the boundary itself

```typescript
// JWT Authentication
function useAuth() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Check if token exists
    const token = localStorage.getItem('token');
    if (token) {
      // Verify token
      fetch('/api/auth/verify', {
        headers: { Authorization: `Bearer ${token}` }
      })
        .then(r => r.json())
        .then(data => {
          setUser(data.user);
          setLoading(false);
        })
        .catch(() => {
          localStorage.removeItem('token');
          setLoading(false);
        });
    } else {
      setLoading(false);
    }
  }, []);

  const login = async (email: string, password: string) => {
    const res = await fetch('/api/auth/login', {
      method: 'POST',
      body: JSON.stringify({ email, password })
    });
    const { token, user } = await res.json();

    // Store in HttpOnly cookie (secure) or localStorage
    localStorage.setItem('token', token);
    setUser(user);
  };

  const logout = () => {
    localStorage.removeItem('token');
    setUser(null);
  };

  return { user, loading, login, logout };
}
```

### 16.16 Error Handling & Recovery

```typescript
// Advanced error handling
class APIError extends Error {
  constructor(
    public statusCode: number,
    message: string,
    public data?: any
  ) {
    super(message);
  }
}

async function fetchData(url: string) {
  try {
    const res = await fetch(url);

    if (!res.ok) {
      throw new APIError(
        res.status,
        `API Error: ${res.statusText}`,
        await res.json()
      );
    }

    return await res.json();
  } catch (error) {
    if (error instanceof APIError) {
      // Handle API errors specifically
      if (error.statusCode === 401) {
        // Refresh token
        await refreshToken();
        return fetchData(url); // Retry
      }
      if (error.statusCode === 429) {
        // Rate limited - wait and retry
        await new Promise(r => setTimeout(r, 5000));
        return fetchData(url);
      }
    }

    // Log to error tracking
    Sentry.captureException(error);
    throw error;
  }
}

// Retry logic
async function retryFetch(url: string, retries = 3) {
  for (let i = 0; i < retries; i++) {
    try {
      return await fetch(url).then(r => r.json());
    } catch (error) {
      if (i === retries - 1) throw error;
      await new Promise(r => setTimeout(r, 1000 * Math.pow(2, i))); // Exponential backoff
    }
  }
}
```

```typescript
// Request deduplication
let pendingRequests: Map<string, Promise<any>> = new Map();

async function fetchWithDedup(url: string) {
  // Return pending request if exists
  if (pendingRequests.has(url)) {
    return pendingRequests.get(url);
  }

  // Create new request
  const promise = fetch(url)
    .then(r => r.json())
    .finally(() => pendingRequests.delete(url));

  pendingRequests.set(url, promise);
  return promise;
}

// Advanced caching strategy
class CacheManager {
  private cache: Map<string, { data: any; expiry: number }> = new Map();

  set(key: string, data: any, ttl: number = 5 * 60 * 1000) {
    this.cache.set(key, {
      data,
      expiry: Date.now() + ttl
    });
  }

  get(key: string) {
    const item = this.cache.get(key);
    if (!item) return null;

    if (Date.now() > item.expiry) {
      this.cache.delete(key);
      return null;
    }

    return item.data;
  }

  clear() {
    this.cache.clear();
  }
}
```

1. ✅ **JavaScript Fundamentals** - Closures, Event Loop, Async/Await
2. ✅ **Browser Internals** - CRP, Reflow/Repaint, Storage
3. ✅ **React Core** - Fiber, Reconciliation, Hooks
4. ✅ **Performance** - Memoization, Bundle Optimization, Virtualization
5. ✅ **State Management** - Redux, Zustand, Context
6. ✅ **Next.js** - SSR/SSG/ISR, Hydration, Caching
7. ✅ **Security** - XSS, CSRF, JWT, CORS
8. ✅ **TypeScript** - Generics, Utility Types, Type Guards
9. ✅ **Testing** - Unit, Integration, E2E tests
10. ✅ **Machine Coding** - Implement common patterns
11. ✅ **System Design** - Design components and systems at scale
12. ✅ **Routing** - Both React Router and Next.js
13. ✅ **API Security** - Input validation, sanitization
14. ✅ **GraphQL** - Basics and advantages over REST
15. ✅ **CMS** - Headless CMS with Strapi
16. ✅ **Top 20 Questions** - Most commonly asked in interviews

**Recommended Study Schedule:**

- **Week 1-2**: JavaScript fundamentals (closures, event loop, promises)
- **Week 3**: React core concepts (Fiber, reconciliation, hooks)
- **Week 4**: Performance optimization and state management
- **Week 5**: Next.js and server-side rendering
- **Week 6**: Security, TypeScript, and testing
- **Week 7**: Machine coding problems
- **Week 8**: System design and mock interviews

**Good luck with your interviews! 🚀**
